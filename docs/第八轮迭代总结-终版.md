## 1. 第八轮迭代新增

本轮为项目真正终轮，聚焦"**Goby 风格网站截图 + Web 爬虫系统 + 管理员操作日志 + L2 插件真实化 + 报告导出增强**"，共涉及 **22 个文件变更**（新增 11 个，修改 11 个）。

### 1.1 网站截图功能升级（Goby 风格）

原截图服务仅支持 `wkhtmltoimage` / `cutycapt` 两种 Linux 工具，Windows 下基本不可用。

**升级为三级回退方案**：

| 优先级 | 方法 | 技术 | 适用场景 |
|:---:|------|------|---------|
| 1 | Playwright | Node.js 无头 Chromium | JS 渲染的现代页面（SPA、动态内容） |
| 2 | wkhtmltoimage | Qt WebKit | 传统服务端渲染页面 |
| 3 | cutycapt | CutyCapt | Linux 环境 fallback |

**自动截图捕获**：在扫描流水线的 Web 指纹识别阶段，对每个发现的 Web 服务自动截取 `http://ip:port` 截图，无需用户手动触发。

**文件变更**：
- `serverscout/backend/scripts/screenshot.mjs`（**新文件**）— Playwright 截图脚本，含 25 秒超时、network idle 等待、多浏览器 UA
- [ScreenshotService.java](serverscout/backend/src/main/java/com/serverscout/service/ScreenshotService.java) — 重写，Playwright 优先 + 三级回退 + `findScreenshot()` 文件查找
- [ScreenshotController.java](serverscout/backend/src/main/java/com/serverscout/controller/ScreenshotController.java) — 新增 `GET /api/v1/screenshot/file/{filename}` 文件服务端点
- [ScanExecutionService.java](serverscout/backend/src/main/java/com/serverscout/service/ScanExecutionService.java) — Web 探针阶段自动截图

### 1.2 管理员操作日志系统

**新建完整的审计日志体系**，与已有的 `VulnStatusLog`（漏洞状态变更日志）互补，覆盖全系统操作。

**数据模型**：[OperationLog.java](serverscout/backend/src/main/java/com/serverscout/entity/OperationLog.java)

| 字段 | 说明 |
|------|------|
| userId, username | 操作用户 |
| operationType | LOGIN_SUCCESS / LOGIN_FAILED / API_CALL |
| target, detail | 操作目标与详情 |
| ipAddress, userAgent | 客户端 IP 和浏览器标识 |
| requestMethod, requestUri | HTTP 方法和路径 |
| statusCode, durationMs | 响应状态码和耗时 |
| createdAt | 操作时间 |

**核心机制**：
- **登录日志**：`AuthController` 的 login/register 方法中记录每次登录尝试（成功/失败），含 IP、User-Agent
- **API 调用日志**：[ApiLoggingFilter.java](serverscout/backend/src/main/java/com/serverscout/config/ApiLoggingFilter.java) 自动拦截所有 POST/PUT/DELETE 及错误请求
- **异步存储**：`@Async` 异步写入，不阻塞业务请求
- **自动清理**：`@Scheduled(cron = "0 0 3 * * ?")` 每天凌晨 3 点清理 90 天前的日志
- **权限控制**：`@PreAuthorize("hasRole('ADMIN')")` 仅管理员可查看

**前端**：[OperationLogViewer.tsx](serverscout/frontend/src/components/OperationLogViewer.tsx)
- 用户名筛选 + 类型下拉（登录成功/登录失败/API调用）
- 彩色类型标签（绿=登录成功、红=登录失败、蓝=API调用）
- IP 地址展示 + 时间轴
- 分页查询

### 1.5 Web 爬虫系统（阶段一：静态爬虫）

**技术选型**：Jsoup（HTML 解析）+ OkHttp（HTTP 客户端），对 Java 生态友好，无需额外进程。

**核心特性**：

| 特性 | 实现 |
|------|------|
| URL 提取 | 解析 `<a href>`、`<link href>` 标签 |
| 同源策略 | 仅爬取同一 host:port |
| 去重 | ConcurrentHashMap 布隆过滤器 + URL 规范化（排序参数、去尾部斜杠） |
| 内容过滤 | Content-Type 白名单（text/html）+ 二进制扩展名过滤 |
| 字符集检测 | Content-Type header → HTML meta 标签 → UTF-8 默认 |
| 请求控制 | User-Agent 轮换（3 个现代 UA）+ 随机延迟（500-800ms） |
| 并发控制 | 线程池（max 3）+ 总页数限制（50） |
| 深度控制 | 可配置最大深度（3） |

**数据模型**：[CrawledUrl.java](serverscout/backend/src/main/java/com/serverscout/entity/CrawledUrl.java)
- 关联 asset、port、task
- 存储 url、path、httpStatus、title、bodyText、linksFound、crawlDepth、responseTimeMs、screenshotPath

**扫描流水线集成**：
```
Nmap 端口扫描 → 资产入库 → 子域名枚举 → Web 指纹识别
                                              ↓
                                   【新增】爬虫发现 (50-57%)
                                              ↓
                                   Nuclei 漏洞扫描 → CVE 匹配 → SSL 采集 → 完成
```

**前端展示**：[CrawledUrlsPanel.tsx](serverscout/frontend/src/components/CrawledUrlsPanel.tsx)
- **截图画廊**（Goby 风格）：水平滚动缩略图、HTTP 状态码叠加层、可点击全屏查看
- **URL 列表**：按深度分组、可展开详情（路径/Content-Type/链接数/响应时间/页面文本摘要）
- **空状态引导**："运行一次带 Web 指纹识别的扫描任务，系统将自动爬取 Web 服务页面并截取网站截图"
- **演示数据**：42 条预置爬虫 URL，覆盖 12 个资产的 Web 端口

**文件变更**（7 个）：
- `entity/CrawledUrl.java`（新）
- `repository/CrawledUrlRepository.java`（新）
- `service/CrawlerService.java`（新，~300 行）

## 5. 遇到的困难与教训

### 5.1 历史困难（第 1-7 轮，共 16 项）

| # | 困难 | 教训 |
|---|------|------|
| 1 | Spring Boot CommandLineRunner 执行顺序 | `@Order` 越小越先执行，多 Runner 显式声明优先级 |
| 2 | JPA LazyInitializationException | LAZY 实体读取必须加 `@Transactional` |
| 3 | React useEffect 无限渲染循环 | 依赖使用 primitive 值，配合 useRef 防重 |
| 4 | Windows `where` 命令多语言输出 | 跨平台 CLI 须处理中/英文错误信息 |
| 5 | Java 8 vs 17 版本冲突 | 多 JDK 环境须显式指定 JAVA_HOME |
| 6 | CORS 端口不匹配 | 前端端口变更时同步更新后端 CORS |
| 7 | 数据隔离关联链复杂度 | `createdBy` 必须在项目第一天就强制赋值 |
| 8 | 验证码原子性问题 | 读题和提交不可分两次网络请求 |
| 9 | `@Scheduled` 无法运行时修改 Cron | 可配置定时任务必须用 `TaskScheduler` |
| 10 | SseEmitter(0L) 内存泄漏 | SSE 长连接必须设超时 + 定期驱逐 |
| 11 | Vite manualChunks 空 chunk | 只配置实际被 import 的依赖 |
| 12 | L2 插件系统安全边界 | 自定义命令即 RCE，需声明风险 |
| 13 | 暗色模式 Tailwind `dark:` 变体覆盖不完整 | 暗色模式需从项目初期建立强制规范，后补成本极高 |
| 14 | Nuclei 在 Windows MSYS2/MinGW 环境下的执行问题 | 区分 shell 环境和 JVM 环境对可执行文件的兼容性 |
| 15 | Censys/VirusTotal API 需 API 密钥的前置依赖 | 第三方 API 集成应提供免费 fallback + 需密钥的高级模式 |
| 16 | 三个折线图在数值接近时完全重叠 | 多系列图表必须用不同线型+面积填充+图例标记区分 |

### 5.2 本轮新困难与教训（第 8 轮，共 5 项）

**17. iText `Cell` 与 Apache POI `Cell` 类名冲突**

- 问题：`com.itextpdf.layout.element.Cell` 和 `org.apache.poi.ss.usermodel.Cell` 同时存在，`Table` 类型也冲突，编译器报"The type Cell/Table is ambiguous"
- 解决：移除所有冲突类的 import，在代码中使用完全限定名 `com.itextpdf.layout.element.Cell`、`com.itextpdf.layout.element.Table`
- 教训：当两个第三方库有同名类时，永远不要用通配符 import（`import com.itextpdf.layout.element.*`），必须显式指定每个类或使用完全限定名

**18. 爬虫演示数据不可见——空状态即隐藏面板**

- 问题：`CrawledUrlsPanel` 在无数据时直接 `return null`，用户完全看不到功能入口
- 解决：改为始终渲染面板，添加空状态引导文案和功能图标说明；同时预置 42 条演示爬虫数据
- 教训：新功能必须有"空状态"设计——告诉用户这个功能是什么、如何触发、预期结果。`return null` 让功能成为幽灵

**19. DemoDataInitializer 数据播种时机问题**

- 问题：`assetRepository.count() >= 5` 的守卫条件阻止了爬虫数据的增量播种。当资产数据已存在但爬虫表为空时，种子逻辑永远不会运行
- 根因：`count()` 查询在 `@Transactional` 下可能受 Hibernate 缓存影响，且表可能尚未被 DDL 创建
- 解决：用 try-catch 包裹 `count()` 调用并记录日志；将爬虫数据播种逻辑从资产播种中解耦，独立检查 `crawledUrlRepository.count() == 0`
- 教训：`CommandLineRunner` 中的条件播种应层层解耦——每个模块独立检查自己的表是否为空，而非依赖其他模块的状态

**20. JPA 懒加载序列化导致 API 返回 500**

- 问题：`CrawledUrl` 的 `@ManyToOne(fetch = LAZY)` 关联（Asset/Port/ScanTask）在 Jackson 序列化时触发 `LazyInitializationException`，API 返回"服务器内部错误"
- 解决：给三个 `@ManyToOne` 关联添加 `@JsonIgnore`，并在类上加 `@JsonIgnoreProperties({"hibernateLazyInitializer", "handler"})`
- 教训：任何包含 JPA 关联的新实体，必须在编写时就决定序列化策略——要么 `@JsonIgnore` 忽略关联，要么使用 DTO 投影，要么 JOIN FETCH 预加载

**21. Java 进程残留导致端口占用**

- 问题：多次 `taskkill` / `Stop-Process` 后仍有机率残留 Java 进程占用 8080 端口，导致新后端启动失败
- 根因：`mvn spring-boot:run` 和 `java -jar` 的后台进程在 bash 脚本中被 kill 后，子进程可能未被正确终止
- 解决：每次启动前执行 `powershell -Command "Get-Process java | Stop-Process -Force"`，并用 `netstat -ano | grep ":8080"` 确认端口释放
- 教训：Windows 下多进程管理的自动化脚本必须包含"验证-清理-再验证"的闭环，不能假设 kill 命令一定成功

### 5.3 过程经验总结（终版，新增 3 条）

1. **暗色模式是贯穿全局的约束**：每新增一个组件都应同步添加 `dark:` 变体
2. **前端 UI 是后端功能的"最后一公里"**：后端 API 必须有前端入口才算真正"可用"
3. **分页系统应尽早统一**：三个列表页各自复制粘贴 → 提取为单一组件
4. **第三方 API 集成要设计降级路径**：免费 API → 需密钥的高级 API → 本地数据库
5. **工具路径检测需要分层搜索**：PATH → 常见安装目录 → 用户手动配置
6. **Cron 表达式对普通用户不友好**：闹钟式时间选择器更直观
7. **配置外置要形成闭环**：三级优先级 + 前端配置 UI + 运行时即时生效
8. **CI/CD 应在项目初期搭建**：后补 CI 虽然可行，但会错过早期捕捉回归问题的机会
9. **`createdBy` 字段必须在第一天就强制赋值**：数据隔离后补成本是初期实现的 10 倍以上
10. **内存泄漏修复需要主动防御**：不能仅依赖框架回调，必须加上定期清理作为安全网
11. **新功能必须有"空状态"设计**：`return null` 让功能成为幽灵，用户根本不知道它的存在
12. **JPA 实体序列化必须在编写时就决策**：`@JsonIgnore` / DTO / JOIN FETCH 三选一，后补成本是初期的 5 倍
13. **多进程管理要形成闭环**：kill → 验证 → 再清理，不能假设命令一定成功
