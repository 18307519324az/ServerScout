## 1. 第七轮迭代新增

本轮为项目终轮，聚焦"暗色模式全面修复 + 用户体验打磨 + 外部情报增强 + 分页系统 + 风险资产 + 工具链完善"，分两次变更集完成，共涉及 **22 个文件变更**（新增 1 个，修改 21 个）。

### 1.2 扫描工具配置增强（Issue #2, #3）

**问题**：
- 工具路径只显示程序名（如 `nmap`），不显示真实完整路径
- 没有检测状态指示，用户不知道工具是否已安装
- 没有安装教程或下载链接

**修复**：

| 层 | 文件 | 变更 |
|----|------|------|
| 后端 | [SystemConfigController.java](serverscout/backend/src/main/java/com/serverscout/controller/SystemConfigController.java) | 重写 `detectTool()` 方法为两层检测：（1）PATH 检测（`where`/`which`）；（2）扫描常见安装目录（Windows: `C:\Program Files (x86)\Nmap\`、`%HOME%\go\bin\`；Unix: `/usr/bin/`、`/opt/homebrew/bin/`、`~/go/bin/`）。新增 `detectFromPath()` 和 `detectFromCommonPaths()` 辅助方法 |
| 前端 | [SettingsPage.tsx](serverscout/frontend/src/pages/SettingsPage.tsx) | 工具路径旁显示检测状态（✅ 已检测到 / ❌ 未检测到）；新增"安装指南"展开面板，包含 Nmap 和 Nuclei 的各平台安装命令与官方下载链接 |

### 1.3 移除冗余扫描配置（Issue #3）

**问题**：系统设置页的"扫描配置"区域提供了创建扫描表单 + 最近扫描列表，与扫描任务列表页（`ScanTaskListPage`）功能完全重复。

**修复**：将"扫描配置"区域（含创建表单、预设模板、最近扫描列表）替换为简洁的"扫描任务管理"入口卡片，链接到 `/scan-tasks` 页面，消除功能冗余。

### 1.4 Cron 表达式 → 闹钟式时间选择器（Issue #4）

**问题**：定时扫描配置使用 Cron 表达式（如 `0 0 2 * * ?`），普通用户难以理解和修改。

**修复**：完全移除 Cron 输入框，替换为直观的下拉选择器：
- **每日巡检**：小时（00-23）+ 分钟（00/15/30/45）下拉选择器
- **每周全面巡检**：小时 + 分钟 + 星期（周一~周日）下拉选择器
- **显示格式**：从 `0 0 2 * * ?` 变为 `每天 02:00`、`每周日 03:00`
- **后端兼容**：前端选择器值在前端自动构建 Cron 字符串传给后端 API，后端无需改动

## 5. 遇到的困难与教训（完整版）

### 5.1 历史困难（第 1-6 轮，共 12 项）

| # | 困难 | 教训 |
|---|------|------|
| 1 | Spring Boot CommandLineRunner 执行顺序 | `@Order` 越小越先执行，多 Runner 显式声明优先级 |
| 2 | JPA LazyInitializationException | LAZY 实体读取必须加 `@Transactional` |
| 3 | React useEffect 无限渲染循环 | 依赖使用 primitive 值，配合 useRef 防重 |
| 4 | Windows `where` 命令多语言输出 | 跨平台 CLI 须处理中/英文错误信息 |
| 5 | Java 8 vs 17 版本冲突 | 多 JDK 环境须显式指定 JAVA_HOME |
| 6 | CORS 端口不匹配 | 前端端口变更时同步更新后端 CORS |
| 7 | 数据隔离关联链复杂度 | `createdBy` 必须在项目第一天就强制赋值 |
| 8 | 验证码原子性问题 | 读题和提交不可分两次网络请求 |
| 9 | `@Scheduled` 无法运行时修改 Cron | 可配置定时任务必须用 `TaskScheduler` |
| 10 | SseEmitter(0L) 内存泄漏 | SSE 长连接必须设超时 + 定期驱逐 |
| 11 | Vite manualChunks 空 chunk | 只配置实际被 import 的依赖 |
| 12 | L2 插件系统安全边界 | 自定义命令即 RCE，需声明风险 |

### 5.3 本轮新困难与教训（第 7 轮，共 4 项）

**13. 暗色模式下 Tailwind `dark:` 变体覆盖不完整**

- 问题：大量组件的 `bg-white`、`text-gray-700` 没有对应的 `dark:bg-gray-800`、`dark:text-gray-200`，导致暗色模式下内容完全不可见
- 解决：逐页审计所有 `bg-white`、`text-gray-*` 出现处，确保每个都有对应的 `dark:` 变体；在全局 CSS 层增加 form 元素的默认暗色覆盖规则
- 教训：暗色模式不是"加一个 toggle 就完事"——必须从项目初期建立强制规范（如 ESLint 规则禁止裸写 `bg-white` 而不配 `dark:bg-*`），后补成本极高

**14. Nuclei 在 Windows MSYS2/MinGW 环境下的执行问题**

- 问题：通过 Go 安装的 `nuclei.exe` (PE32+ x86-64) 在 MSYS2 bash 和 PowerShell 沙箱中均无法执行，报"Exec format error"或"不是有效的应用程序"
- 根因：MSYS2/MinGW bash 无法直接运行原生 Windows PE 可执行文件，需要 cmd.exe 或直接双击运行
- 解决：后端 Spring Boot 运行在 JVM 上，通过 `ProcessBuilder` 可以直接调用 Windows 原生可执行文件，不受 MSYS2 限制
- 教训：在 Windows 混合环境（MSYS2/MinGW/bash）下开发时，需区分 shell 环境和 JVM 环境对可执行文件的兼容性差异

**15. Censys/VirusTotal API 需 API 密钥的前置依赖**

- 问题：Censys（需要 API ID + Secret）和 VirusTotal（需要 API Key）的 API 调用在密钥未配置时只能返回错误提示
- 解决：在系统设置 Webhook 编辑表单中增加 API 密钥输入字段；在查询失败时展示配置引导和 API 密钥获取链接
- 教训：第三方 API 集成应至少提供两种模式——（1）免费无需密钥的 API（如 Shodan InternetDB）作为默认 fallback；（2）需要密钥的高级 API 作为可选增强

**16. 三个折线图在数值接近时完全重叠**

- 问题：30 天趋势图的三条折线均为实线、同宽、无面积填充，在数值相近时完全无法区分
- 解决：采用三种不同线型（solid/dashed/dotted）、三种不同符号（circle/diamond/triangle）、三个不同颜色的渐变面积填充
- 教训：多系列图表在数据特性未知时（可能全为 0 或重叠），必须在设计阶段就做好区分——不同线型 + 面积填充 + 图例标记是最低成本的方案

### 5.4 过程经验总结（终版）

1. **暗色模式是贯穿全局的约束**：不是单个组件的特性，而是整个应用的基准要求。每新增一个组件都应同步添加 `dark:` 变体
2. **前端 UI 是后端功能的"最后一公里"**：定时扫描、Webhook、外部情报等后端早已完成，但缺乏前端入口或仅有占位符导致功能"不可用"
3. **分页系统应尽早统一**：三个列表页各自复制粘贴分页逻辑 → 提取为单一 `<Pagination>` 组件，这是软件工程的基本纪律
4. **第三方 API 集成要设计降级路径**：免费 API（Shodan InternetDB）→ 需要密钥的高级 API（Censys/VirusTotal）→ 本地数据库 fallback
5. **工具路径检测需要分层搜索**：PATH → 常见安装目录 → 用户手动配置，逐层 fallback
6. **Cron 表达式对普通用户不友好**：即使用户是安全运维人员，闹钟式时间选择器也更直观
7. **配置外置要形成闭环**：三级优先级（DB > 环境变量 > 默认值）+ 前端配置 UI + 运行时即时生效
8. **CI/CD 应在项目初期搭建**：后补 CI 虽然可行，但会错过早期捕捉回归问题的机会
9. **`createdBy` 字段必须在第一天就强制赋值**：数据隔离后补成本是初期实现的 10 倍以上
10. **内存泄漏修复需要主动防御**：不能仅依赖框架回调，必须加上定期清理作为安全网
