# 风险评分模型

## 概述

RiskScoreService 实现了基于 5 因子加权评分算法的风险量化模型。每个资产独立计算风险评分，结果限制在 0-100，对应 5 个风险等级。

## 评分公式

```
finalRiskScore =
assetExposureScore × 25%
+ vulnerabilitySeverityScore × 30%
+ serviceRiskScore × 15%
+ exploitabilityScore × 15%
+ businessImportanceScore × 15%
− remediationDeduction
```

最终结果通过 `clamp(0, 100)` 限制在 0-100 之间。

## 子分数说明

### 1. assetExposureScore — 资产暴露面评分（权重 25%）

基于资产开放端口数量评估暴露程度：

| 开放端口数 | 评分 |
|-----------|------|
| 0 | 0 |
| 1-2 | 15 |
| 3-5 | 30 |
| 6-10 | 50 |
| 11-20 | 70 |
| 21+ | 90 |

### 2. vulnerabilitySeverityScore — 漏洞严重性评分（权重 30%）

基于资产上所有开放状态漏洞的严重等级累加：
- 每个严重（CRITICAL）漏洞 +35 分
- 每个高危（HIGH）漏洞 +20 分
- 每个中危（MEDIUM）漏洞 +10 分
- 每个低危（LOW）漏洞 +3 分

总分上限 100，超出则截断。

### 3. serviceRiskScore — 服务风险评分（权重 15%）

基于资产开放端口的类型评估风险，取最高分：

| 端口类型 | 评分 | 示例端口 |
|----------|------|----------|
| 数据库端口 | 85 | 3306(MySQL)、5432(PostgreSQL)、6379(Redis) |
| 远程管理端口 | 75 | 22(SSH)、23(Telnet)、3389(RDP) |
| Web 管理端口 | 70 | 8080、8443、9090 |
| Web 服务端口 | 40 | 80、443 |
| 其他端口 | 20 | — |
| 无端口 | 0 | — |

### 4. exploitabilityScore — 可利用性评分（权重 15%）

基于资产上漏洞的可利用性评估：

| 条件 | 评分 |
|------|------|
| 存在 CRITICAL 漏洞 | 85 |
| 存在 HIGH 漏洞（无 CRITICAL） | 60 |
| 存在含 CVE ID 的漏洞 | 35 |
| 存在漏洞但无 CVE | 10 |
| 无漏洞 | 0 |

### 5. businessImportanceScore — 业务重要性评分（权重 15%）

基于资产的主机名、标签和开放 Web 服务评估业务重要性。基础分 20，以下情况累加：

| 条件 | 加分 |
|------|------|
| 主机名/标签含"production"、"prod"、"core"、"db"等核心关键词 | +40 |
| 主机名/标签含"test"、"dev"、"staging"等测试关键词 | −15 |
| 开放了 HTTP/HTTPS Web 服务 | +5/个，上限 +15 |

总分通过 `clamp(0, 100)` 限制。

### 6. remediationDeduction — 修复扣减

从加权总分中减去修复扣减分，总分 ≤30：

| 条件 | 扣减 |
|------|------|
| 存在有修复建议的 CVE | +10 |
| 每个开放漏洞 | +2（最多 5 个 × 2 = +10） |

## 风险等级划分

| 分数范围 | 风险等级 | 含义 |
|----------|----------|------|
| 0-20 | INFO | 信息，常规资产 |
| 21-40 | LOW | 低风险，建议关注 |
| 41-60 | MEDIUM | 中风险，需要处理 |
| 61-80 | HIGH | 高风险，尽快修复 |
| 81-100 | CRITICAL | 严重风险，立即修复 |

## 为什么无漏洞资产也会有风险分

即使资产没有漏洞，以下因素仍会产生风险评分：

1. **暴露面**：开放大量端口本身就增加了攻击面
2. **服务风险**：暴露了数据库端口（3306/6379）等高危服务
3. **业务重要性**：核心生产资产即使无漏洞也应标注风险

因此一台无漏洞但开放了 MySQL 和 Redis 端口的生产服务器仍会获得 MEDIUM 级别的风险评分。

## Demo Mode 下的风险评分

Demo 模式在扫描完成后自动调用 `RiskScoreService.calculateForTask()` 为该任务生成评分。Demo 数据同样经过完整的评分计算流程，包括 5 因子加权和修复扣减。Demo 扫描任务的风险评分与真实扫描任务完全一致，保证了 Demo 模式下评分展示的真实性。

## Recalculate 事务保证

`recalculate` 接口用于重新计算某个扫描任务下所有资产的风险评分：

```java
@Transactional
public List<RiskScoreDetail> recalculateForTask(Long taskId) {
    riskScoreRepository.deleteByTaskId(taskId);  // 先删除旧数据
    return calculateForTask(taskId);              // 再重新计算
}
```

`@Transactional` 保证 delete + calculate 在同一个数据库事务中执行。如果 calculate 失败，delete 也会回滚，避免出现"评分被清空但未重算成功"的数据丢失问题。

### 为什么 Controller 不加 @Transactional

Controller 层的 `recalculate` 方法只调用了 `riskScoreService.recalculateForTask(taskId)`，不涉及跨多个 Service 的事务协调。将事务注解放在 Service 层而非 Controller 层是 Spring 官方推荐的做法：

- Service 层是业务逻辑的事务边界
- Controller 层负责请求/响应编排，不应承担事务职责
- Service 层 `@Transactional` 更易于单元测试

## 修复建议生成

修复建议程序化生成，包含两个维度：

1. **高危端口建议**：按端口类型生成针对性的安全加固建议
   - SSH (22)：限制白名单 IP 访问，使用密钥认证替代密码
   - RDP (3389)：限制管理 IP，启用 NLA 认证
   - MySQL (3306)：限制访问来源，使用强密码
   - Redis (6379)：配置 requirepass，禁用危险命令

2. **漏洞修复建议**：每个 CVE 生成对应的修复建议
   - 优先使用 CveDatabase 中的 `fixSuggestion` 字段
   - 若无修复建议，使用"升级 {affectedSoftware} 至最新版本"兜底

## 评分示例

### 示例 1：无漏洞 Web 服务器

- 2 个端口（80, 443）：暴露面 15
- 0 漏洞：漏洞严重性 0，可利用性 0
- Web 服务：服务风险 40
- 通用主机名：业务重要性 20

```
raw = 15×0.25 + 0×0.30 + 40×0.15 + 0×0.15 + 20×0.15 = 3.75+0+6+0+3 = 12.75
final = clamp(13−0) = 13 → INFO
```

### 示例 2：含严重漏洞的生产数据库服务器

- 12 个端口（含 3306, 22, 80, 443...）：暴露面 70
- 2 个 CRITICAL + 1 个 HIGH 漏洞：漏洞严重性 90
- 含 MySQL 端口：服务风险 85
- 含 CRITICAL 漏洞：可利用性 85
- 生产服务器：业务重要性 75

```
raw = 70×0.25 + 90×0.30 + 85×0.15 + 85×0.15 + 75×0.15 = 17.5+27+12.75+12.75+11.25 = 81.25
deduction = 10(有修复建议) + 6(3个开放漏洞×2) = 16
final = clamp(81−16) = 65 → HIGH
```

## 接口

```http
# 查询任务的风险评分
GET /api/v1/risk-scores/task/{taskId}

# 查询资产的风险评分
GET /api/v1/risk-scores/asset/{assetId}

# 查询风险排行 Top N
GET /api/v1/risk-scores/top?limit=5

# 重新计算任务风险评分（幂等）
POST /api/v1/risk-scores/task/{taskId}/recalculate
```

所有接口需要 JWT 认证。
