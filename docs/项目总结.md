# ServerScout 项目总结

> 服务器资产攻击面可视化分析平台  
> 日期：2026-05-20（第三轮修复版）  
> 技术栈：React 18 + TypeScript + Vite + Tailwind CSS / Spring Boot 3.3 + Java 17 + MySQL 8.0

---

## 一、项目概述

ServerScout 是一个轻量级服务器资产攻击面可视化分析平台，面向中小型企业安全运维人员，提供从资产发现、端口扫描、服务指纹识别、子域名枚举到漏洞关联的全流程攻击面管理能力。

核心定位：**对标 Goby/FOFA 的攻击面测绘能力，轻量易部署、可视化优先、可扩展、聚焦 Web 资产**。

## 五、扫描引擎

### 扫描策略

| 类型 | Nmap 参数 | 说明 |
|------|----------|------|
| quick | `-sS --top-ports 100 -sV --version-intensity 3` | SYN 扫描 Top 100 + 轻量版本识别 + Banner 抓取 |
| full | `-sS -sV -O --version-intensity 7` | 全端口 + 深度版本识别 + OS + 多脚本采集 |

### 完整扫描流水线

```
0-5%    启动 Nmap 端口扫描
5-30%   端口扫描执行中 (解析进度)
30-40%  资产入库 + 去重 (Port 同步 + Banner 存储)
40-44%  子域名枚举 (crt.sh 证书透明度 + DNS 爆破, 仅域名目标)
44-55%  Web 指纹识别 (310 规则 + favicon hash + 响应头匹配)
55-57%  Nuclei 漏洞扫描
57-70%  CVE 匹配 (基于 Nmap 产品名 + Web 指纹 CMS/框架/WAF 名)
70-85%  SSL 证书采集 (TLS 握手 → 证书链 → SAN → 过期检测)
85-90%  证书透明度分析 (SSL SAN → crt.sh 子域名枚举)
90-100% 任务完成
```

## 六、漏洞检测体系

### 三层漏洞发现

| 层级 | 方式 | 说明 |
|------|------|------|
| Layer 1: Nuclei | 基于模板的漏洞扫描 | 对 HTTP 服务 URL 格式扫描 |
| Layer 2: CVE 匹配 | 服务指纹 → CVE 数据库匹配 | 基于 Nmap 产品名 + Web 指纹名 |
| Layer 3: CVE 种子库 | 115 条预置 CVE | 覆盖 Web 服务器、CMS、Java 框架、OA 系统、安全设备 |

### CVE 匹配过滤机制

- 排除 30+ 通用服务名（http, https, ssh, ftp, dns 等）
- 排除 OS 级 CVE（Microsoft Windows, Linux Kernel, Sudo, glibc 等）
- 仅匹配 ≥3 字符的具体产品名
- 同 CVE + 同资产去重

## 八、第三轮遇到的困难与教训（新增）

### 困难 1: G6 v5 渲染彻底失败

G6 v5 创建了 4 层 Canvas 元素（背景层、主层、标签层、覆盖层），但 `graph.render()` 后采样了画布上 16 个不同位置的像素点，全部为透明（alpha=0）。节点从未被实际绘制。尽管尝试了调整力导向参数（`nodeStrength`, `collideStrength`, `linkDistance`, `maxIteration`）、强制清除旧 Canvas、使用 `initRef` 防止双重初始化，问题始终无法解决。

**最终方案**：放弃 Canvas 方案，用原生 SVG + 自实现力导向布局替代。SVG 节点在 DOM 中可见、可查询、可点击，彻底避免了"渲染了但看不到"的黑盒问题。

**教训**：Canvas 图形库在出现渲染问题时极难调试 — 你无法 "检查元素" 查看节点是否在那里，只能采样像素。对于数据可视化需求不极端（<100 个节点）的场景，SVG 的透明性和可调试性远优于 Canvas。

### 困难 2: React StrictMode 导致双重副作用

React 18 开发模式下的 StrictMode 会 Mount→Unmount→Re-mount 组件两次，导致 `useEffect` 执行两次。用 `useRef` 加锁方案看似可行，但在 Unmount/Re-mount 周期中 ref 会被重置，导致锁失效。

**最终方案**：在 `useEffect` 清理函数中彻底清除所有 DOM 节点（`while (container.firstChild) container.removeChild(container.firstChild)`），确保无论执行多少次，容器始终保持清洁。

### 困难 3: Spring Data Page JSON 序列化格式变化

Spring Boot 3.3.x 中 `Page` 对象序列化为 JSON 时，`totalElements`、`totalPages` 等字段被嵌套在 `page` 子对象中，而非与 `content` 并列。与前端 `PageData` 类型定义的顶层字段不匹配。

**教训**：前后端接口对接时应优先用真实 API 响应验证数据结构，而非依赖文档或经验假设。Spring Boot 3.x 的 `spring.jackson.serialization` 配置会影响 Page 序列化格式。

### 困难 4: 前端依赖管理的脆弱性

一次文件误操作导致 `node_modules` 损坏，Vite CLI 二进制丢失，页面 CSS 全部失效。误删后恢复的唯一可靠方式是完整重建 `node_modules`。

**教训**：`node_modules` 应始终被 `.gitignore` 排除，但 `package-lock.json` 必须纳入版本控制。恢复时 `npm ci` 比 `npm install` 更可靠（严格按 lock 文件安装）。

### 困难 5: GitHub 推送权限反复受阻

两个 Fine-grained PAT 都缺少 `Contents: write` 权限，导致 403。只有 Classic PAT 的 `repo` scope 才能正常推送。此外，Windows 环境下 Git 的 SSL 证书验证也导致了一次 `SSL certificate problem` 错误。

**教训**：GitHub 的 Fine-grained PAT 权限模型比 Classic PAT 更细粒度但也更容易配错。推送代码优先使用 Classic PAT 或 `gh` CLI（GitHub Desktop 已预装但 git-bash 中未配置 `gh` 命令）。

## 十、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 认证方式 | JWT 无状态 | 适合前后端分离，无需服务端 Session |
| 扫描引擎 | 策略模式 + @Async | 便于扩展新扫描器，异步不阻塞 |
| 实时推送 | SSE (SseEmitter) | 简单够用，无需 WebSocket 复杂性 |
| 数据库 | MySQL 8.0 | 开发环境可用 |
| 状态管理 | TanStack Query | 自动缓存、失效重取、轮询 |
| 图表 | ECharts 5 | 统计图首选 |
| **拓扑图** | **原生 SVG + 力导向** | 替代 @antv/g6 v5 — Canvas 渲染失败后选择 SVG 透明可控 |
| PDF | iText 8 | Java 生态最成熟的 PDF 库 |
| 子域名枚举 | crt.sh + DNS 爆破 | 零外部 API Key 依赖 |
| 指纹匹配 | mmh3 Favicon Hash | 与 FOFA/EHole 兼容 |
| CVE 过滤 | 白名单+黑名单 | 排除通用服务和 OS 级 CVE |

## 十二、安全注意事项

- JWT Secret 和数据库密码硬编码在 `application.yml` 中，生产部署需使用环境变量
- 扫描功能调用外部 CLI 工具 (Nmap/Nuclei)，需确保这些工具已安装且在 PATH 中
- 漏洞扫描结果中的 CVE 链接指向 `nvd.nist.gov`，仅供参考
- 本工具仅用于授权安全测试，不得用于未授权的网络扫描
- GitHub Token 不应出现在代码中，本项目的 Token 仅用于一次性的仓库创建和推送
