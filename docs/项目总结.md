# ServerScout 项目总结

> 服务器资产攻击面可视化分析平台  
> 日期：2026-05-20（更新版）  
> 技术栈：React 18 + TypeScript + Vite + Tailwind CSS / Spring Boot 3.3 + Java 17 + MySQL 8.0

---

## 一、项目概述

ServerScout 是一个轻量级服务器资产攻击面可视化分析平台，面向中小型企业安全运维人员，提供从资产发现、端口扫描、服务指纹识别、子域名枚举到漏洞关联的全流程攻击面管理能力。

核心定位：**对标 Goby/FOFA 的攻击面测绘能力，轻量易部署、可视化优先、可扩展、聚焦 Web 资产**。

## 五、扫描引擎（增强版）

### 扫描策略

| 类型 | Nmap 参数 | NSE 脚本 | 说明 |
|------|----------|---------|------|
| **quick** | `-sS --top-ports 100 -sV --version-intensity 3` | `--script banner --script-timeout 30s` | SYN 扫描 Top 100 + 轻量版本识别 + Banner 抓取 |
| **full** | `-sS -sV -O --version-intensity 7` | `--script banner,http-title,ssl-cert,ssl-enum-ciphers --script-timeout 60s` | 全端口 + 深度版本识别 + OS + 多脚本采集 |

### 完整扫描流水线

```
0-5%    启动 Nmap 端口扫描 (含 NSE banner/http-title/ssl-cert 脚本)
5-30%   端口扫描执行中 (解析 --stats-every 进度)
30-40%  资产入库 + 去重 (Port 同步 + Banner 存储)
40-44%  子域名枚举 (crt.sh 证书透明度 + DNS 爆破, 仅域名目标)
44-55%  Web 指纹识别 (310 规则 + favicon hash + 响应头/Cookie/正则匹配)
55-57%  Nuclei 漏洞扫描 (enableVulnScan=true, 自动添加 http:// 前缀)
57-70%  CVE 匹配 (基于 Nmap 服务产品名 + Web 指纹 CMS/框架/WAF 名)
70-85%  SSL 证书采集 (TLS 握手 → 证书链 → SAN → 过期检测)
85-90%  证书透明度分析 (SSL SAN → 提取域名 → crt.sh 子域名枚举)
90-100% 任务完成
```

## 六、漏洞检测体系

### 三层漏洞发现

| 层级 | 方式 | 说明 |
|------|------|------|
| **Layer 1: Nuclei** | 基于模板的漏洞扫描 | 对 HTTP 服务使用 `http://IP` URL 格式扫描 |
| **Layer 2: CVE 匹配** | 服务指纹 → CVE 数据库匹配 | 基于 Nmap 产品名 + Web 指纹 CMS/框架/WAF 名 |
| **Layer 3: CVE 种子库** | 115 条预置 CVE | 覆盖 Web 服务器、CMS、Java 框架、OA 系统、安全设备 |

### CVE 匹配过滤机制

- **排除通用服务名**：http, https, ssh, ftp, dns, tcpwrapped 等 30+ 个
- **排除 OS 级 CVE**：Microsoft Windows, Linux Kernel, Sudo, glibc, Polkit 等
- **仅匹配具体产品名**：产品名 ≥ 3 字符且不在排除列表
- **去重**：同一 CVE + 同一资产不重复关联

## 八、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 认证方式 | JWT 无状态 | 适合前后端分离，无需服务端 Session |
| 扫描引擎 | 策略模式 + @Async | 便于扩展新扫描器，异步不阻塞 |
| 实时推送 | SSE (SseEmitter) | 简单够用，无需 WebSocket 复杂性 |
| 数据库 | MySQL 8.0 | 开发环境可用 JSON 类型 |
| 状态管理 | TanStack Query | 自动缓存、失效重取、轮询 |
| 图表 | ECharts 5 | 统计图首选 |
| **拓扑图** | **@antv/g6 v5** | v5 力导向布局 + Canvas 高性能渲染，**替换 D3.js** |

| **子域名枚举** | **crt.sh + DNS 爆破** | 零外部 API Key 依赖，被动+主动结合 |

## 九、主要 Bug 修复记录

### Bug #1: Hibernate LazyInitializationException（反复出现）

| 出现位置 | 原因 | 修复 |
|----------|------|------|
| `probeWebServices()` → `port.getAsset().getIpAddress()` | `@Async` 方法中事务已关闭，Port.asset 懒加载代理无法初始化 | `HttpProbeService.probePort()` 改为接收 IP 参数 |
| `collectSslCerts()` → `port.getAsset().getIpAddress()` | 同上 | `SslCertService.fetchCertificate()` 改为接收 IP 参数 |
| `AssetService.getAssetDetail()` → `v.getCveDatabase().getCveId()` | `@Transactional` 缺失，Session 在方法返回前关闭 | 添加 `@Transactional(readOnly = true)` |
| `VulnerabilityService.getVulnerabilityDetail()` | 同上 | 添加 `@Transactional(readOnly = true)` |
| `matchCves()` → `mapping.getAsset().getIpAddress()` | `ScanAssetMapping.asset` 懒加载 | 新增 `findByScanTaskIdWithAsset` (JOIN FETCH) |

**教训**: Spring Boot `@Async` + JPA 懒加载的组合非常容易出问题。`@Async` 方法中每个子步骤都应该有独立的 `@Transactional`，或者使用 JOIN FETCH 一次性加载所有需要的关联实体。JPA 的 `spring.jpa.open-in-view: false` 配置使问题更加突出。

### Bug #2: @antv/g6 v5 API 兼容性

| 现象 | 原因 | 修复 |
|------|------|------|
| 拓扑图渲染后无内容 | G6 v5 `behaviors` 必须是对象数组 `{type:'drag-canvas'}`，v4 的字符串数组 `['drag-canvas']` 不工作 | 全部改为对象格式 |
| 图初始化后不显示 | G6 v5 自动渲染，不需要也不能调用 `graph.render()` | 移除 render() 调用 |
| 空数据无提示 | 无节点时 graph 区域全白，用户不知发生了什么 | 添加空状态 UI "暂无拓扑数据" |

**教训**: 前端库的主版本号升级往往有大量 API 破坏性变更。G6 从 v4 到 v5 的变化没有在 package.json 中体现（直接装了 v5），需要查阅 v5 文档而非依赖 v4 记忆。

### Bug #3: Java var 类型推断失败

| 现象 | 原因 | 修复 |
|------|------|------|
| `java.lang.Error: Unresolved compilation problems` | `var wf = repository.findByPortId(...).orElse(null)` → 编译器无法从 `null` 推断类型 | 改为显式类型 `Optional<WebFingerprint> wfOpt` |

**教训**: Java `var` 在与 `Optional.orElse(null)` 组合使用时类型推断会退化到 `Object`，导致后续方法调用编译失败。涉及 null 的场景应使用显式类型。

### Bug #4: CVE 误报过多

| 现象 | 原因 | 修复 |
|------|------|------|
| 扫描 47.120.72.205 发现 57 个 critical 漏洞 | `matchCves()` 对 `serviceName="http"` 做 `LIKE '%http%'` 匹配，命中所有包含 "http" 的 CVE | 1. 排除 30+ 个通用服务名；2. 排除 18 类 OS 级 CVE；3. 仅匹配 ≥3 字符的具体产品名 |
| 匹配到 Microsoft/Linux 内核漏洞 | CVE 种子包含 OS 级漏洞，且匹配未区分 | 添加 `SKIP_CVE_AFFECTED` 黑名单过滤 |

**教训**: CVE 匹配的精度与召回率权衡是一个核心难点。过于宽松会产生大量误报，过于严格会漏报。多层过滤（通用服务 → OS 级 → 具体产品名）是较优方案。

### Bug #5: 扫描端口范围错误

| 现象 | 原因 | 修复 |
|------|------|------|
| 输入 `47.120.72.205/80` 期望只扫 80 端口，实际扫描了 1-1000 | `/80` 被当作 CIDR 掩码通过验证，端口范围默认 `1-1000` | 1. 拒绝掩码 >32 的非法 CIDR；2. 支持 `IP:端口` 格式自动提取；3. 表单增加提示文案 |

## 十、遇到的困难与经验教训

### 困难 1: GitHub 资源无法直接访问

网络环境限制使 `gh` CLI 和 `raw.githubusercontent.com` 均不可用。替代方案是通过 WebSearch 查找指纹库格式描述，手动构建 310 条规则。如果能够直接拉取 hfinger/FingerprintHub 等开源指纹库，规则数量和质量还能进一步提升。

### 困难 2: @Async + @Transactional 的组合陷阱

Spring 的 `@Async` 会切换线程，`@Transactional` 默认只对当前线程生效。在 `executeScan()` 这个 `@Async` 方法中，内部调用 `this.probeWebServices()` 不走代理，即使加了 `@Transactional` 也不生效。最终选择在 Repository 层用 `JOIN FETCH` 解决。

### 困难 3: G6 v5 无报错静默失败

G6 v5 在 `behaviors` 格式错误时不会抛出异常或控制台错误，只是图形区域保持空白。调试了一轮 API 差异才定位到问题。如果 G6 能在初始化时对不支持的参数格式给出 warning，会节省大量排查时间。

### 困难 4: CVE 匹配的精度控制

初始实现过于激进——对任何检测到的服务名做模糊 LIKE 匹配，导致每个资产关联数十个不相关的 CVE。改进过程中逐步加入了多层过滤：通用服务名排除 → OS 级 CVE 黑名单 → 产品名长度阈值。精度与召回率的平衡仍需在实际使用中根据反馈持续调整。

### 经验教训

1. **JPA 项目应始终使用 `spring.jpa.open-in-view=false`** —— 但必须意识到这意味着所有懒加载访问必须包裹在 `@Transactional` 内
2. **@Async 方法内部的数据库访问要格外小心** —— 涉及关联实体的查询应使用 JOIN FETCH 一次性加载
3. **前端库升级大版本前应查阅 migration guide** —— v4→v5 的 API 变化是拓扑图问题的根本原因
4. **编译错误的逃逸** —— Java 的 `var` 类型推断失败在某些 IDE 中可能不报错，但运行时抛出 `java.lang.Error`，CI/CD 的 `mvn compile` 是最后防线
5. **安全扫描工具的 CVE 匹配需要持续的规则维护** —— 不是一次性工作，需要根据误报反馈不断调整过滤规则
6. **指纹规则的数量是竞争力的核心** —— Goby/FOFA 等商业产品有几千条规则，310 条仅覆盖了主要产品，后续可通过对接 FingerprintHub 等社区源持续扩充
