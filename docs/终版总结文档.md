## 一、项目概述

ServerScout 是一个轻量级服务器资产攻击面可视化分析平台，面向中小型企业安全运维人员，提供从资产发现、指纹识别、漏洞关联到报告导出的全流程攻击面管理能力。

**技术栈**: React 18 + TypeScript + Vite + Tailwind CSS + Spring Boot 3.3.5 + Java 17 + MySQL 8.0

### 2.4 SSE 内存泄漏修复

**之前状态**：`SseEmitter(0L)` 永不超时，`ProgressEmitter` 无定期清理机制，若扫描异常挂起则 emitter 永不释放。

| 修复项 | 描述 | 文件 |
|--------|------|------|
| 30 分钟超时 | `new SseEmitter(30*60*1000L)` 替代无限制的 `0L` | [ProgressEmitter.java](serverscout/backend/src/main/java/com/serverscout/service/ProgressEmitter.java) |
| 时间戳追踪 | `TimedEmitter` 内部类记录创建时间，支持过期判断 | 同上 |
| 显式清理 | `cleanup(taskId)` 方法供扫描取消时调用 | 同上 |
| 定期驱逐 | `@Scheduled(fixedRate=600000)` 每 10 分钟清理超过 1 小时的僵尸 emitter 条目 | 同上 |
| 空列表回收 | `removeEmitter` 中列表为空时自动从 Map 移除 | 同上 |

## 六、遇到的困难与教训

### 6.1 历史困难（前五轮）

| # | 困难 | 教训 |
|---|------|------|
| 1 | Spring Boot CommandLineRunner 执行顺序 | `@Order` 数值越小越先执行；多 Runner 依赖时显式声明所有优先级 |
| 2 | JPA LazyInitializationException | 涉及 LAZY 实体读取的 Runner 必须加 `@Transactional` |
| 3 | React useEffect 无限渲染循环 | useEffect 依赖使用 primitive 值，配合 useRef 防重 |
| 4 | Windows `where` 命令多语言输出 | 跨平台 CLI 工具须处理中/英文错误信息 |
| 5 | Java 8 vs 17 版本冲突 | 多 JDK 环境须显式指定 JAVA_HOME |
| 6 | CORS 端口不匹配 | 前端端口变更时同步更新后端 CORS |
| 7 | 数据隔离关联链复杂度 | `createdBy` 必须在项目第一天就强制赋值 |
| 8 | 验证码原子性问题 | 读题和提交不可分两次网络请求 |

### 6.2 本轮新困难与教训

**9. @Scheduled 注解无法运行时修改 Cron**

- 问题：`@Scheduled(cron = "${...}")` 在 Spring 启动时解析一次即固定，无法运行时动态修改
- 解决：弃用注解，改用 `TaskScheduler.schedule(Runnable, CronTrigger)` 实现动态调度
- 教训：需要运行时可配置的定时任务，必须用编程式 `TaskScheduler` 而非声明式 `@Scheduled`

**10. SseEmitter(0L) 潜在的永久内存泄漏**

- 问题：`new SseEmitter(0L)` 永不超时，若客户端异常断连或扫描挂起，emitter 永不释放
- 解决：设置 30 分钟超时 + TimedEmitter 时间戳追踪 + 10 分钟定期驱逐
- 教训：SSE 长连接必须设置合理的超时时间，并配合定期清理僵尸连接

**11. Vite manualChunks 空 chunk 问题**

- 问题：`@antv/g6` 和 `d3` 在代码中未被实际引用（历史依赖），导致 manualChunks 生成 0KB 空文件
- 解决：从 manualChunks 配置中移除未使用的库
- 教训：manualChunks 只配置实际被 import 的依赖，定期清理 package.json 中的无效依赖

**12. L2 插件系统的安全边界**

- 问题：自定义命令扫描插件本质上是任意命令执行（RCE），在生产环境中是严重安全隐患
- 当前措施：仅 ADMIN 角色可管理插件，插件命令通过 ProcessBuilder 执行
- 教训：插件系统应增加命令白名单校验或沙箱隔离，毕业设计阶段以功能演示为主但需在文档中声明风险

### 6.3 过程经验总结

1. **前端 UI 是后端功能的"最后一公里"**：定时扫描和 Webhook 后端早已完成，但缺乏前端配置入口导致功能"不可用"
2. **内存泄漏修复需要主动防御**：不能仅依赖 onCompletion/onTimeout 回调，必须加上定期驱逐作为安全网
3. **ECharts 按需加载收益显著**：1MB 的图表库独立分包后，首屏体积从 ~1.5MB 降至 ~480KB（gzip ~135KB）
4. **配置外置要形成闭环**：三级优先级（DB > 环境变量 > 默认值）是生产环境的最佳实践
5. **审计日志是安全合规的基线要求**：漏洞状态变更的记录不仅用于审计，也可用于后续的趋势分析
6. **CI/CD 应在项目初期搭建**：后补 CI 虽然可行，但会错过早期捕捉回归问题的机会

## 九、启动方式

### 环境要求
- Java 17+, MySQL 8.0+, Node.js 18+, Maven 3.8+

### 后端启动
```bash
cd serverscout/backend
mvn clean package -DskipTests
java -jar target/server-scout-1.0.0.jar
```

### 前端启动
```bash
cd serverscout/frontend
npm install
npm run dev
```

- 管理员：admin / admin123
