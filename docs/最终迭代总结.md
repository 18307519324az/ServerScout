### 1.2 多用户数据隔离 — 关键安全修复

**问题**：所有用户登录后看到的仪表盘、资产、扫描任务完全相同，存在严重数据泄露风险。

**根因**：
1. `ScanTask.createdBy` 字段在创建任务时从未被赋值
2. 所有 Service 层使用全局 `findAll()` / `count()` 查询
3. Controller 层未将认证用户信息传递给 Service

**解决方案**：通过 `ScanTask.createdBy` → `Asset.task` → `Port/WebFingerprint/Vulnerability` 关联链实现数据隔离。ADMIN 角色用户可查看全部数据，USER 角色用户仅能查看自己创建的扫描任务及其关联数据。

## 4. 遇到的困难与教训

### 4.1 困难 1：Java 8 vs Java 17 版本冲突

**现象**：`mvn spring-boot:run` 报 ClassNotFoundException，但 `mvn compile` 成功。

**根因**：系统 PATH 中 `java` 指向 JDK 8，Spring Boot 3.3.5 需要 JDK 17。

**解决**：必须使用完整路径 `"C:/Program Files/Java/jdk-17/bin/java" -jar target/server-scout-1.0.0.jar` 显式指定 JDK 17。

**教训**：在 Windows 多 JDK 环境下，包装启动脚本是必要的。或直接设置 `JAVA_HOME` 指向 JDK 17。

### 4.2 困难 2：CORS 端口不匹配导致 403

**现象**：前端登录返回 403，但 curl 测试正常。

**根因**：CORS 配置只允许 `localhost:5173`，实际 Vite 因端口被占用启动在 `5174`。

**解决**：`WebConfig` 添加 `5174` 到 allowedOrigins。

**教训**：前端端口变更时需同步更新后端 CORS 白名单。

### 4.3 困难 3：Maven 编译与文件锁

**现象**：修改 Java 文件后 `mvn package` 报错 `Unable to rename JAR to JAR.original`。

**根因**：运行中的 JAR 被 JVM 进程锁定，Maven repackage 插件无法重命名。

**解决**：先 `taskkill /F /IM java.exe` 杀掉所有 Java 进程，再 `mvn package`。

**教训**：重建前务必停止运行中的 JAR；开发时可用 `mvn compile` + `spring-boot:run` 避免 repackage 步骤。

### 4.4 困难 4：Playwright MCP ref selector 不可用

**现象**：Playwright MCP 工具的快照使用 `ref=e45` 格式的引用，但 `browser_click`/`browser_type` 等工具不支持 `ref=` 选择器引擎。

**根因**：MCP 工具的 Playwright 版本未注册 ref 引擎，ref 引用仅用于快照中的可读标识。

**解决**：使用 `browser_run_code_unsafe` 直接编写 Playwright 代码，用 `placeholder` 属性选择器精确定位表单元素。

**教训**：在 MCP 限制场景下，使用 `page.$()` 和 `page.fill()` 等标准 Playwright API 是可靠的 fallback。

### 4.5 困难 5：注册测试中验证码时间窗口问题

**现象**：填写完注册表单后提交，多次显示"操作失败"。

**根因**：分两次 `browser_run_code_unsafe` 调用：第一次读取验证码题目，第二次提交答案。两次调用之间验证码可能已刷新（或因网络延迟、状态变更触发）。

**解决**：将所有操作合并到单次 `browser_run_code_unsafe` 调用中，确保验证码读取和提交在同一 JavaScript 事件循环内完成。

**教训**：验证码这种有时效性的状态，读取和使用必须在同一个原子操作中完成。

### 4.6 困难 6：数据隔离改造的关联链复杂度

**现象**：单个功能（如仪表盘统计）需要 JOIN 3-4 张表才能实现用户过滤。

**根因**：数据模型的设计链路是 `ScanTask → Asset → Port/Vulnerability/WebFingerprint`，每个 Repository 都需要手工编写 JPQL 多表 JOIN 查询。

**解决**：在每个 Repository 中手写 `@Query` JPQL，通过 `t.createdBy = :createdBy` 参数化过滤。为每个需要过滤的查询方法编写对应的 `*ByCreatedBy` 变体。

**教训**：
1. 项目初期应设计 `createdBy` 字段并强制赋值，后补成本远高于初期
2. 可考虑使用 Spring Security 的 `@PostFilter` 注解简化代码，但性能开销需评估
3. 更优雅的方案是将用户上下文注入到 Repository 层（如 Spring Data JPA AuditorAware），避免每个方法签名都携带 username

### 4.7 本轮教训总结

1. **注册体验对标主流网站**：姓名/性别/邮箱是用户系统的基本标配，初期就应支持
2. **数据隔离是安全基线**：多用户系统必须在第一天就考虑数据隔离，否则是严重的数据泄露风险
3. **`createdBy` 字段必须在创建任务的第一天就赋值**：后补需要同时修复 4 层（Entity → Repository → Service → Controller）
4. **admin 看全部、普通用户看自己**是最简单有效的权限模型
5. **JPQL 多表 JOIN 手写是体力活**：如有 Spring Data JPA Specifications 或 QueryDSL，可减少重复代码
6. **E2E 测试应在每次改动后立即执行**：早发现早修复，避免多轮迭代后积累大量回归问题
7. **验证码操作必须原子化**：读题和提交不可分两次网络请求
