# 面试讲解

## 一分钟项目介绍

ServerScout 是一个基于 Spring Boot 和 React 构建的服务器资产攻击面管理与风险分析平台，集成了 Nmap 和 Nuclei，用于完成资产发现、端口与服务识别、漏洞检测、攻击面可视化、报告导出和 AI 风险简报生成。这个项目主要体现我在后端服务设计、前后端协作、扫描工具集成和安全数据展示方面的工程实践能力。

## 三分钟项目介绍

项目的核心目标是把分散的安全扫描结果整合成统一平台。后端使用 Spring Boot，负责扫描任务创建、扫描编排、结果聚合、JWT 认证、统一异常处理和报告导出；前端使用 React + TypeScript，负责资产列表、漏洞列表、拓扑可视化、报告中心和 AI 风险简报等界面展示。

在扫描能力上，我通过 Nmap 完成端口与服务探测，通过 Nuclei 完成漏洞模板检测，再把结果和 CVE、CVSS、EPSS 等信息结合，形成可查看、可导出、可解释的风险数据。

为了提高项目的可演示性，我实现了三个特色设计：

1. **Demo Mode**：不依赖真实 Nmap/Nuclei，用模拟数据跑通完整流程，现场演示零配置
2. **阶段状态机**：9 阶段 Pipeline + 实时进度追踪，展示细粒度的扫描过程可视化
3. **风险评分模型**：5 因子加权评分，每个资产获得 0-100 的量化风险分数和可解释的风险原因

项目还支持 PDF / Excel 报告导出、威胁情报查询和 AI 风险简报生成。

## 整体架构怎么讲

从外到内分层介绍：

1. **前端层**：React 18 + TypeScript + Vite，使用 `@tanstack/react-query` 管理 API 状态
2. **API 网关层**：Nginx / Vite Proxy 代理，JWT 认证过滤
3. **Controller 层**：RESTful 接口，统一 `ApiResponse<T>` 响应结构
4. **Service 层**：业务逻辑核心，策略模式实现扫描引擎，独立服务管理风险评分、报告导出
5. **数据访问层**：JPA / Hibernate，MySQL 持久化，Redis 缓存
6. **外部集成**：Nmap、Nuclei、威胁情报 API、AI API

## 扫描流程怎么讲

```text
创建扫描任务 → 校验目标 → 策略选择（Demo/Real）
→ Demo: DemoScannerStrategy 生成模拟数据
→ Real: Nmap 端口扫描 → Nuclei 漏洞检测
→ CVE 匹配 → 风险评分 → 结果保存 → 前端展示 → 报告导出
```

重点说明：使用策略模式（ScannerStrategy 接口），不同扫描策略可插拔替换，不影响主流程。

## Demo Mode 怎么讲

Demo Mode 通过环境变量 `SCANNER_DEMO_MODE=true` 开启（默认开启），`DemoScannerStrategy` 实现 `ScannerStrategy` 接口：

- `supports()` 检查 demoMode 状态，为 true 时支持所有扫描类型
- `execute()` 按扫描类型生成 2-8 个资产、2-8 个端口、0-8 个漏洞
- IP 地址根据目标范围智能衍生，端口从预置池选取，漏洞引用 CveDatabase 已有 ID

**为什么不做简单 Mock？** 因为 Demo Mode 不只是模拟数据，它还完整走了 9 阶段 Pipeline，触发真实的风险评分计算，生成可导出报告。这样演示者可以在零配置环境下展示完整的业务闭环。

## 阶段状态机怎么讲

扫描任务分为 9 个阶段，每个阶段有 5 种状态：

```
目标校验 → 端口扫描 → 服务识别 → Web 探测 → 漏洞检测
→ CVE 匹配 → 风险分析 → 结果保存 → 通知回调
```

- 每个阶段记录开始时间、结束时间、耗时、错误信息
- 前端通过轮询获取阶段列表，实时展示进度
- 失败阶段不阻塞已保存的数据（如端口扫描失败不影响已发现的资产）
- 风险分析阶段失败不会清除已有的资产和漏洞数据

## 风险评分模型怎么讲

5 因子加权评分：

```
final = exposure×25% + severity×30% + serviceRisk×15%
        + exploitability×15% + businessImportance×15% − remediationDeduction
```

0-100 分对应 INFO/LOW/MEDIUM/HIGH/CRITICAL 五个等级。

**设计重点**：

- 无漏洞资产也有分（暴露面+服务风险+业务重要性）
- 修复建议自动生成（端口加固 + CVE 修复）
- Recalculate 使用 `@Transactional` 保证删除+重算原子性
- 评分结果存储在 `risk_score_detail` 表，支持按任务/资产查询和排行

## 报告导出怎么讲

- **Excel**：Apache POI，7 个 Sheet（概览/资产/端口/漏洞/阶段/评分/修复建议）
- **PDF**：iText 7，含概览、阶段、风险 Top 10、高危漏洞、修复建议、安全声明
- 中文通过 `PDF_FONT_PATH` 环境变量指定字体文件
- Demo Mode 下报告标记为"演示数据"

## Docker 部署怎么讲

三容器架构：MySQL 8.0 + Spring Boot + Nginx。

```bash
docker-compose up -d --build  # 一键启动
```

- 后端 Dockerfile 多阶段构建，包含 Nmap + Nuclei + 中文字体
- 前端 Dockerfile 构建 Nginx 静态资源 + API 反向代理
- MySQL 数据持久化到 named volume

## 异常处理怎么讲

- 统一响应结构 `ApiResponse<T>`（code + message + data）
- `GlobalExceptionHandler` 捕获各类异常，返回统一格式
- `BusinessException` 携带业务错误码（40000 参数错误、50000 服务器错误等）
- 认证异常返回 401 JSON，权限不足返回 403 JSON
- 扫描工具异常分级封装，不暴露底层细节

## 并发与安全边界怎么讲

- JWT Token 认证，密码 BCrypt 加密 + RSA 加密传输
- Admin/USER 两级角色权限
- 操作日志审计关键动作
- Demo Mode 不执行真实扫描，不扫描公网目标
- 扫描目标格式校验，不接收非法目标

## 项目难点

### 1. 扫描工具集成与解耦

Nmap 和 Nuclei 输出格式完全不同，需要在 Service 层统一抽象。解决方案：策略模式，`ScannerStrategy` 接口定义 `supports()` 和 `execute()`，不同的扫描策略各自实现自己的解析逻辑。

### 2. 扫描进度实时追踪

扫描异步执行，前端需要实时看到进度。解决方案：9 阶段状态机，每个阶段独立记录状态，前端轮询获取。Demo Mode 通过 sleep 模拟各阶段时间，真实模式由扫描工具实际执行进度驱动。

### 3. 风险评分量化

漏洞等级不能直接等于风险分。解决方案：5 因子加权模型，综合考虑暴露面、漏洞严重性、服务风险、可利用性和业务重要性，加上修复扣减。

### 4. Recidive 数据一致性问题

重新计算评分时需要先删除旧数据再重新计算，如果删除后计算失败，数据就丢了。解决方案：`@Transactional` 将删除和重算放在同一事务，任一失败整体回滚。

### 5. 中文字体渲染

PDF 导出中文需要额外的字体文件配置，不同环境路径不同。解决方案：通过环境变量 `PDF_FONT_PATH` 配置，Docker 镜像预装 Noto Sans CJK。

### 6. 多阶段构建优化

为了减小 Docker 镜像体积，后端采用 Maven 多阶段构建：build 阶段使用 Maven 镜像编译，runtime 阶段只复制 Jar 包和运行环境。

## 项目亮点

1. **不依赖 CRUD 模板**：项目有明确的业务闭环 — 扫描 → 分析 → 评分 → 报告
2. **可演示性**：Demo Mode 让项目在零环境下也可以完整演示
3. **工程能力外化**：阶段状态机、风险评分模型、报告导出等设计体现系统性思考
4. **安全工具集成深度**：Nmap/Nuclei 不只是简单 shell 调用，做了结果结构化、CVE 关联、风险评级
5. **全栈覆盖**：后端（Spring Boot + JPA + Security + 报表）、前端（React + TypeScript + ECharts）、DevOps（Docker + CI）

## 项目不足（诚实版）

1. 工具链依赖本地环境安装，跨平台一致性仍需提升
2. 更复杂的任务调度与插件体系仍可继续完善
3. 部分模块的自动化测试覆盖还可以继续补强
4. 扫描效率在大规模目标下需要进一步优化
5. AI 风险简报在无 LLM 接口时只能做规则分析，生成质量有限

## 面试官可能追问

### 1. 为什么要做 Demo Mode？

**答**：项目面向毕业设计和面试展示场景。现场演示时通常没有 Nmap/Nuclei 环境，也缺少可扫描的授权目标。Demo Mode 用模拟数据跑通完整流程，让演示者在任何环境下都能展示项目的全栈能力。同时 Demo Mode 的开发也帮我验证了扫描策略接口的抽象是否合理。

### 2. 为什么 Demo Mode 不执行真实 Nmap / Nuclei？

**答**：两个原因。一是环境依赖问题 — 演示环境不一定安装了这些工具；二是安全边界 — Demo Mode 在默认配置下开启，如果不加控制地执行真实扫描，可能会在演示时意外扫描到未授权目标。Demo Mode 的设计目标就是"安全演示"。

### 3. 如果 Nmap 不存在怎么办？

**答**：分两种情况。Demo Mode 下不需要 Nmap，系统直接用模拟数据。非 Demo Mode 下如果 Nmap 不存在，扫描任务会失败，标记对应的扫描阶段为 FAILED，并在任务详情中显示错误信息。前后端基础页面不受影响。建议生产环境通过 Docker 部署，镜像中预装了 Nmap。

### 4. 如何防止用户扫描未授权公网目标？

**答**：这个项目明确面向授权资产场景。在代码层面做了以下控制：Demo Mode 默认开启，不执行真实扫描；非 Demo Mode 要求用户显式配置工具路径；前端创建任务时提示安全声明。但需要说明的是，单靠软件层面无法完全阻止恶意使用，安全声明和文档提示了合法使用边界。

### 5. 为什么要设计扫描阶段状态机？

**答**：扫描是一个异步的长时间操作，用户需要知道"现在进行到哪一步了"和"如果失败了是哪个阶段出问题"。阶段状态机解决了这个需求。每个阶段独立跟踪状态、耗时和错误信息，前端可以精确展示进度。Demo Mode 也基于阶段状态机，保证了两种模式下的前后端交互一致性。

### 6. 风险评分为什么不是简单按漏洞等级？

**答**：漏洞等级只是风险的一个维度。一台服务器即使没有漏洞，如果暴露了数据库端口（如 3306、6379）或远程管理端口（如 22、3389），攻击面就已经扩大了。一台业务重要性高的生产服务器即使只有一个 LOW 漏洞，也需要优先处理。所以我的评分模型综合考虑了 5 个维度，让评分结果更贴近真实的安全风险判断。

### 7. recalculate 为什么要加事务？

**答**：`recalculate` 的流程是"先删除旧数据，再重新计算写入"。如果删除成功但计算失败，该任务的风险评分数据就永久丢失了。把这两个操作放在一个 `@Transactional` 中，任何一步失败整体回滚，保证数据不会进入不一致状态。

### 8. 报告导出如何处理中文乱码？

**答**：Excel 导出使用 Apache POI，POI 原生支持 Unicode，中文不需要特殊处理。PDF 导出使用 iText 7，需要额外的中文字体文件（如 Noto Sans CJK）。通过 `PDF_FONT_PATH` 环境变量配置字体路径，Docker 镜像中预装了字体。如果字体配置不正确，PDF 中文会显示为方块，但 Excel 不受影响。

### 9. 这个项目和普通 CRUD 项目区别在哪里？

**答**：普通 CRUD 项目主要是围绕单表的增删改查，技术难点集中在页面交互和数据校验上。ServerScout 的特点是：

1. **业务逻辑复杂**：扫描编排、工具调用、结果解析、CVE 匹配、风险评分，每一层都有非平凡的算法
2. **外部系统集成**：Nmap、Nuclei、威胁情报 API、AI API，需要处理格式差异、超时、异常
3. **状态管理**：扫描任务的异步执行、阶段状态机、实时进度追踪
4. **数据关联深度**：资产 → 端口 → 漏洞 → CVE → 证书 → 指纹，形成完整的资产画像
5. **多维输出**：API + 前端 + PDF + Excel + AI 简报，同一份数据多种呈现方式

这些加在一起，让这个项目的工程复杂度远超普通的 CRUD 应用。
