### 2.2 CVE 漏洞库聚焦 Web 应用

**文件**: `CveDataInitializer.java`（重写）, `ScanExecutionService.java`（matchCves 重写）

#### 问题

旧 CVE 种子数据包含大量 OS 级漏洞（Microsoft Windows、Linux Kernel、sudo、polkit、glibc 等），CVE 匹配逻辑对所有端口做 `LIKE %keyword%` 模糊匹配，导致 85 条漏洞中大部分与 Web 资产无关。

#### 修复

1. **删除 32 条 OS 级 CVE**，精简为 76 条 Web 应用级 CVE（Apache/Nginx/Tomcat/Spring/Struts/WordPress/Drupal/Jenkins/GitLab 等）
2. **匹配逻辑三重过滤**:
   - 仅对 `isWebService=true` 的端口匹配
   - 新增 `isWebAppProduct()` 白名单校验（含 60+ Web 关键词）
   - `SKIP_CVE_AFFECTED` 黑名单（30+ OS 级软件名）
3. **优先使用 Web 指纹数据**（CMS/框架/WAF 名称）进行匹配，其次才是端口服务名
4. **数据库清理**: 删除旧 OS 级 CVE 的 asset_vulnerability 关联记录

## 三、Bug 修复记录

### 3.1 重大 Bug：@Async 不生效（阻塞 41 秒）

**现象**: 点击"开始扫描"后前端卡住 41 秒，对话框无法关闭。

**根因**: Spring AOP 代理不拦截类内部方法调用。`@Async` 只有通过代理 Bean 调用才生效。

**修复**: 新建 `ScanExecutionService`，将 `@Async` 方法移到独立 Bean 中，通过注入调用。

```
修复前: ScanService.createTask() → this.executeScanAsync()  ← 代理不拦截
修复后: ScanService.createTask() → scanExecutionService.executeScan()  ← 代理拦截
```

**教训**: Spring 的 `@Async`、`@Transactional`、`@Cacheable` 等注解都依赖 AOP 代理，绝不能在同一类内通过 `this.xxx()` 调用。

### 3.2 严重 Bug：拓扑图节点点击完全不响应

**现象**: 用户点击拓扑图中的节点，侧边栏不显示任何信息。

**根因分析**（经过 7 次迭代调试）:

| 尝试 | 方法 | 结果 | 原因 |
|------|------|------|------|
| 1 | `graph.on('node:click', ...)` | ❌ | G6 v5 事件不触发 |
| 2 | `graph.on('click', ...)` | ❌ | 同上 |
| 3 | React `onClick` on container div | ❌ | G6 canvas 捕获所有指针事件，阻止冒泡 |
| 4 | React `onPointerDownCapture` on container | ❌ | G6 在 document 层注册了捕获监听，先于 React 拿到事件 |
| 5 | Native `container.addEventListener('pointerdown', handler, true)` | ❌ | G6 canvas 内的事件不传播到容器 |
| 6 | Native `window.addEventListener('pointerdown', handler, true)` | ✅ | window 层捕获早于 G6 |
| 7 | 备选方案：React 组件按钮 | ✅✅ | 100% 可靠，不依赖 canvas |

**最终方案（双通道）**:
1. **主通道**：工具栏「资产列表」按钮 → 纯 React 组件交互 → 100% 可靠
2. **辅助通道**：`window.addEventListener('pointerdown', handler, true)` + `window.addEventListener('click', handler, true)` + `container.addEventListener('click', handler)` → 三层监听 → 坐标命中检测

命中检测逻辑：
```typescript
const nodePos = graph.getElementPosition(nodeId)  // 获取 G6 内部坐标
const dx = clickX - nodePos[0]
const dy = clickY - nodePos[1]
if (√(dx² + dy²) ≤ max(nodeSize, 50)) → 显示侧边栏
```

**耗时**: ~4 小时

**教训**: Canvas 渲染库（G6/D3/ECharts）的事件系统与 React 的集成是高风险区域。应该始终提供纯 UI 交互（按钮/列表）作为备选方案，不要只依赖 canvas 事件。

### 3.3 严重 Bug：拓扑图节点全部重叠成一条线

**现象**: 直接打开拓扑页面，所有节点挤在一起，只显示一条细线。

**根因**: G6 在 modal 的 CSS flex 布局完成前就已初始化，容器 `clientWidth/clientHeight` 为 0，力导向算法在零尺寸空间中计算，所有节点坐标坍缩到一点。

**修复（5 项措施）**:
1. `setTimeout(100ms)` 延迟初始化，等待 flex 布局完成
2. `requestAnimationFrame` 循环（最多 30 次）等待容器获得非零尺寸
3. CSS `min-height` 三级保护：modal 500px / graph area 400px / container 300px
4. 力导向参数增强：`nodeStrength: 1000`, `linkDistance: 350`, `maxIteration: 500`, `collideStrength: 2`
5. 渲染后调用 `graph.fitView()` 居中所有节点

**耗时**: ~1.5 小时

### 3.4 中等 Bug：拓扑图需要 F12 才能正常显示

**现象**: 正常打开拓扑页面看不到图表，但按 F12 打开开发者工具后刷新就能看到。

**根因**: 打开开发者工具改变了 viewport 尺寸 → 触发 window resize 事件 → G6 重新渲染。普通加载缺少这个 resize 信号，如果初始化时容器尺寸为 0，后续没有事件触发重渲染。

**修复**:
- 添加 `ResizeObserver` 监听容器尺寸变化，自动调用 `graph.setSize(w, h)`
- 初始化时用 `requestAnimationFrame` 循环等待尺寸就绪

### 3.5 中等 Bug：仪表盘和所有 API 无法访问

**现象**: 前端所有请求报 `ECONNREFUSED`，仪表盘加载中永远不出来。

**根因**: 后端未启动（编译错误 + JDK 版本不匹配）。

**修复**:
1. 修复 `ScanExecutionService.java` 中 Java Record 访问器错误（`getCms()` → `cmsName()`）
2. 修复 `AssetService.java` 中 Lambda 变量非 effectively final 错误
3. 使用 JDK 17 完整路径启动后端

### 3.6 G6 v5 数据格式不兼容

**现象**: 自定义节点属性（openPortCount、criticalVulnCount 等）在 style 函数中无法读取。

**根因**: G6 v5 要求自定义数据放在 `{ id: '...', data: { ... } }` 的 `data` 子对象中，不能直接平铺在节点对象上。

**修复**: 重构所有节点数据结构，自定义属性移入 `data` 对象，style 函数通过 `d.data?.xxx` 访问。

### 3.7 G6 v5 不自动渲染

**现象**: `new Graph({...})` 创建后 canvas 空白，没有任何渲染内容。

**根因**: 代码注释写"G6 v5 auto-renders, no need to call graph.render()" 是**错误的**。G6 v5.1 必须显式调用 `await graph.render()`。

**修复**: 添加 `await graph.render()`，useEffect 改为 async 模式。

### 3.8 边类型警告洪水

**现象**: 控制台 >150 条 `[G6 v5.1.1] The element service of edge is not registered` 警告。

**根因**: G6 v5 将 edge 的 `type` 字段解释为内部元素类型（line/polyline/cubic），自定义值 "service"/"gateway"/"network" 不是有效类型，每帧渲染都报警告。

**修复**: 将自定义类型移入 `data.linkType`，边样式函数通过 `d.data?.linkType` 读取。

### 3.9 X 按钮跳转 about:blank

**现象**: 拓扑图点击 X 关闭按钮 → 跳转到 `about:blank`。

**根因**: `navigate(-1)` 在无浏览器历史记录时（如直接打开 /topology）跳转到空白页。

**修复**: 改为 `navigate('/dashboard')`。

### 3.10 其他已修复 Bug

| Bug | 文件 | 修复 |
|-----|------|------|
| LazyInitializationException (任务详情 500) | ScanService.java | 加 `@Transactional(readOnly = true)` |
| 资产列表 500 (JPQL LIKE %:keyword%) | AssetRepository.java | 改用 `CONCAT('%', :keyword, '%')` |
| 资产列表序列化 500 | Asset.java | task 字段加 `@JsonIgnore` |

## 五、关键技术决策

### 5.1 G6 v5 的艰难选择

| 维度 | 评价 |
|------|------|
| 渲染性能 | ✅ Canvas 渲染，大数据量表现好 |
| 力导向图 | ✅ 原生支持，参数丰富 |
| v4→v5 兼容 | ❌ API 大量变更，无迁移指南 |
| 事件系统 | ❌ 与 React 集成极其困难 |
| TypeScript | ❌ 类型定义不完整 |
| 文档 | ❌ 示例多为 v4 语法 |

**经验**: 对于小型拓扑图（< 50 节点），**ECharts graph** 或 **D3.js force** 可能是更稳定的选择。如果继续使用 G6 v5，应该在架构上准备好纯 UI 交互备选方案。

### 5.2 双通道交互模式（新架构原则）

```
用户想查看资产详情：
  ┌─────────────────────────────────────────┐
  │ 通道 1（主）：工具栏「资产列表」按钮      │
  │   → React onClick → setSelectedNode()   │
  │   → 100% 可靠，不依赖任何 canvas 事件    │
  ├─────────────────────────────────────────┤
  │ 通道 2（辅）：点击拓扑图节点              │
  │   → window capture listener → 坐标检测  │
  │   → 依赖 G6 getElementPosition() API    │
  └─────────────────────────────────────────┘
```

这个模式应该推广到所有 canvas 密集交互的页面。

## 六、踩坑记录

### 坑 1: Spring @Async 自调用陷阱
- **现象**: 扫描阻塞请求线程 41 秒
- **根因**: `this.executeScanAsync()` 绕过 AOP 代理
- **耗时**: ~2 小时
- **教训**: 注解驱动功能一律通过注入调用

### 坑 2: G6 v5 事件系统与 React 完全不兼容
- **现象**: 节点点击不触发任何事件
- **根因链**: G6 在 document 层注册捕获监听 → 阻止所有常规事件传播 → `graph.on()` API 自身也不工作
- **耗时**: ~4 小时（7 次迭代）
- **最终方案**: window 捕获 + 三层监听 + 坐标命中检测
- **教训**: 评估 canvas 库时优先验证其事件系统与宿主框架的兼容性

### 坑 3: G6 v5 初始化时机
- **现象**: 节点重叠成线 / 需要 F12 刷新
- **根因**: CSS flex 布局未完成 → 容器尺寸为 0 → 力导向在零空间中计算
- **耗时**: ~1.5 小时
- **教训**: Canvas 库必须在 DOM 完全布局（flex/grid 计算完毕）后才能初始化

### 坑 4: G6 v5 不自动渲染
- **现象**: canvas 空白无内容
- **根因**: 官方或社区文档误导说 v5 自动渲染，实际必须显式 `await graph.render()`
- **教训**: 不盲信文档和注释，以实际运行结果为准

### 坑 5: G6 v5 数据格式变更
- **现象**: 自定义属性无法在 style 函数中读取
- **根因**: v5 要求 `{ id, data: { customProps } }` 结构
- **教训**: 升级主版本号时全面检查数据格式

### 坑 6: G6 v5 边 type 字段重载
- **现象**: 150+ 条 "edge is not registered" 警告
- **根因**: `type` 在 v5 中是内部元素类型名，不能用作自定义分类
- **教训**: 框架保留字段不要用作自定义数据

### 坑 7: Hibernate LAZY + Jackson = 500
- **现象**: 资产列表/任务详情间歇性 500
- **根因**: LAZY 加载的代理无法在事务外序列化
- **出现次数**: 3 次（Asset.task、ScanAssetMapping.asset、AssetVulnerability.cveDatabase）
- **解决方案**: `@JsonIgnore` / `@Transactional(readOnly)` 二选一

### 坑 8: Java Record 访问器命名
- **现象**: `pr.getCms()` 编译错误
- **根因**: Java Record 的访问器方法名为字段名本身（`cmsName()` 而非 `getCms()`）
- **教训**: 注意区分 Lombok @Data（生成 getXxx()）和 Java Record（生成 xxx()）

### 坑 9: JDK 版本混乱
- **现象**: `UnsupportedClassVersionError: class version 61.0`
- **根因**: Windows 上安装了 JDK 8 和 JDK 17，PATH 中 JDK 8 在前
- **教训**: 使用 wrapper 脚本或指定完整路径启动

### 坑 10: bash 环境残缺
- **现象**: `find`/`grep`/`sleep`/`curl`/`head` 等命令全部不可用
- **根因**: Windows Git Bash 是精简环境
- **影响**: 大幅降低调试效率
- **应对**: 使用 Python 替代 shell 脚本，使用 Playwright 替代 curl 测试
