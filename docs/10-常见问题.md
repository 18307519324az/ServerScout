# 常见问题

## 安装与启动

### 1. 后端启动报错：数据库连接失败

**错误信息**：`com.mysql.cj.jdbc.exceptions.CommunicationsException: Communications link failure`

**原因**：MySQL 服务未启动或连接配置不正确。

**解决方法**：
1. 确认 MySQL 服务已启动：`net start mysql`（Windows）或 `systemctl start mysql`（Linux）
2. 检查 `application.yml` 中的 `spring.datasource` 配置是否正确
3. 如果使用 Docker，先启动 MySQL 容器：`docker-compose up -d mysql`
4. 等待 MySQL 完全启动后再启动后端（约 10-15 秒）

### 2. 前端启动报错：Node.js 版本不兼容

**错误信息**：`error: The engine "node" is incompatible with this module`

**原因**：项目要求 Node.js 18+，当前版本过低。

**解决方法**：
1. 检查当前版本：`node --version`
2. 升级到 Node.js 18 或更高版本
3. 或使用 nvm 管理多版本：`nvm install 20 && nvm use 20`

### 3. 前端启动后页面空白或 API 请求失败

**原因**：Vite 代理配置不正确或后端未启动。

**解决方法**：
1. 确认后端已启动：`curl http://localhost:8080/actuator/health`
2. 检查 `frontend/vite.config.ts` 中的代理配置，确保 `/api` 指向正确的后端地址
3. 后端默认端口为 8080，如有修改需同步更新代理配置
4. 查看浏览器控制台网络请求，确认请求是否到达后端

### 4. Docker 部署失败：端口冲突

**错误信息**：`port is already allocated`

**原因**：本机已运行 MySQL（3306）或其他服务占用映射端口。

**解决方法**：
1. 修改 `docker-compose.yml` 中的映射端口：
   ```yaml
   ports:
     - "3308:3306"  # 本机 3306 被占用，映射到 3308
   ```
2. 或先停止本机 MySQL：`net stop mysql` / `systemctl stop mysql`

### 5. Docker 部署失败：内存不足

**错误信息**：`Cannot connect to the Docker daemon` 或容器异常退出

**原因**：Docker 可用内存不足（至少需要 2GB）。

**解决方法**：
1. Docker Desktop：Settings → Resources → 增加 Memory 到 4GB+
2. 清理无用容器和镜像：`docker system prune -a`
3. 单独启动服务，不依赖 Docker Compose

## 扫描功能

### 6. Demo Mode 下创建扫描任务后长时间无响应

**原因**：Demo Mode 通过 sleep 模拟各阶段耗时，full 扫描约需 5-8 秒完成。

**解决方法**：
1. 等待 10 秒后刷新页面
2. 检查浏览器控制台是否有网络错误
3. 确认后端日志中是否显示 `Demo mode is active` 相关日志

### 7. 创建扫描任务后状态一直为 "running"

**原因**：扫描异步执行，前端通过轮询获取进度。如果后端处理异常或前端轮询失败，状态可能不更新。

**解决方法**：
1. 等待 15 秒后手动刷新页面
2. 检查后端日志是否有异常堆栈
3. 检查网络请求 `/api/v1/scan-tasks/{id}/stages` 是否正常返回
4. 如果任务卡住，可删除任务重建

### 8. 真实模式下 Nmap 或 Nuclei 执行失败

**错误信息**：后端日志显示 `Nmap execution failed` 或 `Nuclei execution failed`

**原因**：Nmap/Nuclei 未安装或路径配置不正确。

**解决方法**：
1. 确认工具已安装：`nmap --version`、`nuclei -version`
2. 检查 `application.yml` 中的路径配置：
   ```yaml
   app:
     scan:
       nmap-path: /usr/bin/nmap       # Linux/Mac
       nuclei-path: /usr/bin/nuclei
   ```
   Windows 下需指定完整路径：`C:\\Program Files\\Nmap\\nmap.exe`
3. 通过 Docker 部署可避免此问题（镜像预装 Nmap + Nuclei）

### 9. Demo Mode 下是否真的不会执行扫描？

**答**：是的。Demo Mode 开启后，`DemoScannerStrategy` 的 `supports()` 对所有扫描类型返回 `true`，`ScanExecutionService` 会优先选择 Demo 策略，不会调用 Nmap 或 Nuclei。可通过后端日志确认：Demo 模式会输出 `Demo mode is active, using DemoScannerStrategy`。

## 报告导出

### 10. PDF 报告中文显示为方块

**原因**：iText 7 需要额外的中文字体文件，未找到字体时会使用默认字体（不支持中文）。

**解决方法**：
1. 确认字体文件存在：检查 `PDF_FONT_PATH` 指向的文件路径
2. 安装 Noto Sans CJK 字体：
   ```bash
   # Ubuntu/Debian
   apt-get install fonts-noto-cjk
   
   # Docker 镜像已预装
   ```
3. 在 `application.yml` 或环境变量中配置正确的字体路径
4. 切换操作系统语言为中文（Windows 下部分字体路径与 Linux 不同）

### 11. Excel 报告下载后乱码

**原因**：Apache POI 原生支持 Unicode，Excel 乱码通常由以下原因导致：

1. HTTP 响应头 `Content-Type` 未正确设置
2. 浏览器未正确识别文件编码

**解决方法**：
1. 确认请求 URL 正确：`GET /api/v1/reports/excel?taskId={taskId}`
2. Excel 报告不依赖中文字体，中文乱码通常不是字体问题
3. 检查后端日志中是否有 POI 相关异常

### 12. 报告导出按钮灰色不可用

**原因**：只有状态为 `completed` 的扫描任务才能导出报告。

**解决方法**：
1. 确认扫描任务已完成（任务列表中状态为"完成"）
2. 等待扫描完成后再尝试导出
3. Demo Mode 下的已完成任务同样支持报告导出

## 登录与权限

### 13. 登录失败：账号或密码错误

**默认账号**：`admin / admin123`（管理员）、`demo_user / demo123`（普通用户）

**解决方法**：
1. 确认使用正确的默认账号
2. 检查后端启动日志中是否有 `Demo admin user is ready` 提示
3. 如果 Demo Mode 关闭且数据库清空，需手动注册新账号
4. 检查数据库 `sys_user` 表中是否存在 admin 用户

### 14. 登录成功后接口返回 401

**原因**：JWT Token 过期或未正确传递。

**解决方法**：
1. Token 有效期默认为 24 小时，过期需重新登录
2. 检查前端请求头是否包含 `Authorization: Bearer <token>`
3. 清除浏览器本地存储后重新登录
4. 检查前后端时间是否同步（Token 签发时间校验）

### 15. 接口返回 403 Forbidden

**原因**：当前用户角色权限不足。

**解决方法**：
1. Admin 角色可访问所有接口
2. USER 角色部分管理接口受限
3. 使用 admin 账号登录可获取完整权限

## 数据库

### 16. 表结构未自动创建

**原因**：JPA `ddl-auto` 配置不正确或数据库连接失败。

**解决方法**：
1. 确认 `application.yml` 中 `ddl-auto` 设置为 `update`
2. 检查数据库连接配置是否正确
3. 手动执行 `serverscout-init.sql` 初始化表结构和演示数据
4. 检查数据库用户是否有建表权限

### 17. 数据库连接池满

**错误信息**：`HikariPool-1 - Connection is not available, request timed out after 30000ms`

**原因**：数据库连接未正确释放或并发请求过多。

**解决方法**：
1. 重启后端服务
2. 检查慢查询，优化数据库索引
3. 适当调大连接池大小：
   ```yaml
   spring:
     datasource:
       hikari:
         maximum-pool-size: 20
   ```

## 部署与环境

### 18. 如何在生产环境关闭 Demo Mode？

**方法**：
1. 设置环境变量：`SCANNER_DEMO_MODE=false`
2. 或修改 `application.yml`：`app.scan.demo-mode: false`
3. 重启后端服务
4. 验证：`curl http://localhost:8080/api/v1/config/demo-mode` 返回 `{"demoMode": false}`

### 19. 前端如何修改 API 地址？

**开发环境**：修改 `frontend/vite.config.ts` 中的代理目标地址：
```typescript
proxy: {
  '/api': {
    target: 'http://localhost:8080',  // 修改为目标后端地址
    changeOrigin: true,
  }
}
```

**生产环境（Nginx）**：修改 Nginx 配置中的 `proxy_pass` 地址：
```nginx
location /api/ {
    proxy_pass http://backend:8080/;  // 修改为目标后端地址
}
```

### 20. 如何修改 JWT 密钥？

**方法**：
1. 生成新的安全密钥（建议 256 位以上）
2. 设置环境变量：`JWT_SECRET=<新密钥>`
3. 或修改 `application.yml`：`app.jwt.secret: <新密钥>`
4. 重启后端服务
5. **注意**：修改密钥后所有已签发的 Token 立即失效，用户需重新登录

## 其他

### 21. 前端构建警告：TypeScript 类型错误

**解决方法**：
1. 检查 `tsconfig.json` 中的 `strict` 模式设置
2. 修复类型定义或使用 `// @ts-ignore` 临时绕过（不推荐）
3. 运行 `npx tsc --noEmit` 定位具体错误位置

### 22. 如何添加新的威胁情报源？

当前威胁情报查询通过 `ThreatIntelService` 实现，支持扩展：
1. 在 `ThreatIntelService` 中添加新的 API 调用方法
2. 在 `application.yml` 中配置 API 地址和密钥
3. 新增的接口需要在前端对应模块中添加展示

### 23. 日志级别如何调整？

修改 `application.yml`：
```yaml
logging:
  level:
    com.serverscout: DEBUG           # 项目日志
    org.springframework: WARN        # Spring 框架日志
    org.hibernate: WARN              # Hibernate 日志
```

推荐调试时使用 `DEBUG` 级别查看扫描流程详情。

### 24. 操作日志在哪里查看？

**答**：操作日志通过 `OperationLog` 实体记录到数据库 `operation_log` 表，前端可在"系统管理 → 操作日志"页面查看。日志内容包括操作人、操作类型、操作详情、IP 地址和操作时间。

### 26. Nmap 已检测到，为什么扫描还是 Demo 数据？

**答**：Nmap（或 Nuclei）已检测到只代表系统中存在该可执行文件，系统在启动时检测其可用性并报告。但**实际是否执行真实扫描由 `SCANNER_DEMO_MODE` 环境变量决定**。Demo Mode 下即使 Nmap 可用，也不会调用，而是使用模拟数据生成器。你可以通过以下方式确认当前模式：

- 侧边栏底部标签（显示"演示模式"或"真实模式"）
- 设置页的运行模式信息卡片
- 后端 API：`GET /api/v1/system/mode`

### 27. 如何切换到真实扫描？

**答**：
1. 编辑 `.env` 文件，将 `SCANNER_DEMO_MODE=true` 改为 `SCANNER_DEMO_MODE=false`
2. 确保系统中已安装 Nmap 和（可选）Nuclei
3. 重启服务：

```bash
docker compose down
docker compose up -d --build
```

切换后，创建扫描任务时会要求确认目标授权，且报告会标注为真实扫描数据。

### 28. 真实扫描会不会扫描公网目标？

**答**：默认配置下，公网扫描是禁止的（`SCANNER_ALLOW_PUBLIC_TARGETS=false`）。即使改为允许，系统也会在创建任务时要求确认目标授权。请务必遵守安全合规要求，仅扫描已获得明确授权的资产。

### 29. 如何判断某个历史任务是 Demo 扫描还是 Real 扫描？

**答**：每个扫描任务在创建时会记录当时的运行模式到 `scanMode` 字段，该字段不会随系统模式切换而改变。你可以在以下位置查看：

- **任务详情页**：任务名称下方显示"演示数据"（黄色标签）或"真实扫描"（蓝色标签），以及"扫描模式：Demo Mode / Real Mode"
- **报告中心**：每个已完成任务下方显示"数据来源：演示模式 / 真实扫描 / 未记录"
- **报告中**：PDF/Excel 报告的封面和声明区标注了数据来源

`scanMode` 为 `null` 的旧任务会显示"未记录"。

### 25. 如何清空所有演示数据？

**方法**：
1. 方法一（推荐）：关闭 Demo Mode 并重启，演示数据不会被清除，但新扫描将使用真实工具
2. 方法二：执行 SQL 清空相关表：
   ```sql
   DELETE FROM risk_score_detail;
   DELETE FROM vulnerability;
   DELETE FROM asset_port;
   DELETE FROM asset;
   DELETE FROM scan_task_stage;
   DELETE FROM scan_task;
   ```
3. 方法三：删除数据库重建（谨慎操作）：
   ```bash
   docker-compose down -v   # 删除数据卷
   docker-compose up -d     # 重新创建
   ```
