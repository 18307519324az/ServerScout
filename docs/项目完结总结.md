### 1.1 PDF 报告中文字体渲染修复

**问题**：iText 8 生成 PDF 时未嵌入中文字体，导致所有中文显示为空白方格或乱码，报告排版混乱完全不可读。

**修复方案**：
- 添加 `loadChineseFont()` 方法，自动搜索多个系统路径的中文字体（Windows/macOS/Linux）
- 支持 `.ttf` 和 `.ttc` 字体格式（TTC 通过 `path,0` 语法指定集合索引）
- 添加 `cPara()` 辅助方法替代 `new Paragraph()`，自动应用中文字体
- 新增 `app.pdf.font-path` 配置项，支持自定义字体路径
- 候选字体路径：Microsoft YaHei → SimSun → Noto Sans SC → PingFang → Noto Sans CJK

**文件**：
- `ReportService.java` — 全文重构，所有 Paragraph 通过 `cPara()` 创建
- `application.yml` — 新增 `app.pdf.font-path` 配置

**验证**：生成的 PDF 中确认嵌入 `MicrosoftYaHei` 字体子集，中文正常渲染。

### 1.4 扫描任务详情页导出修复

**问题**：扫描任务详情页（`/scan-tasks/:id`）的"导出 PDF 报告"和"导出 Excel 报告"按钮使用了原生 `<a href="...">` 链接。浏览器直接导航时不会携带 JWT `Authorization` header，后端 SecurityConfig 拦截返回 **403 Forbidden → Whitelabel Error Page**。

- 处理函数调用 `downloadPdfReport()` / `downloadExcelReport()`（经过 axios 拦截器自动附加 JWT token）

## 五、开发困难与教训总结

### 5.1 本轮新增困难

#### 难点 6：iText 8 中文字体嵌入

**问题**：iText 8 默认字体 Helvetica 不支持中文，所有中文在 PDF 中显示为空白。升级到 iText 8.0.4 后 API 有较大变化（`PdfFontFactory.createFont()` 签名变更，TTC 字体需要 `path,0` 格式指定集合索引）。

**解决过程**：
1. 排查确认是字体缺失问题（而非编码问题）
2. 编写 `loadChineseFont()` 多路径回退搜索（Windows → macOS → Linux）
3. iText 8 的 `.ttc` 字体不支持 `createFont(path, index, encoding, strategy)` 签名，需使用 `createFont("path,0", encoding, strategy)` 格式
4. 创建 `cPara()` 辅助方法统一管理字体设置
5. 全局替换 `new Paragraph()` → `cPara()`，共约 40 处

**教训**：iText 主要版本（7 → 8）API 有破坏性变更，查看官方文档时应确认版本匹配。PDF 生成应从一开始就考虑中文字体问题，不要等到最后再修。

---

#### 难点 7：蜜罐规则设计

**问题**：如何在"低误报"和"广覆盖"之间平衡。过于严格的规则会漏掉变种蜜罐，过于宽松的规则会产生大量误报。

**解决过程**：
1. 研究主流蜜罐框架（Cowrie、Dionaea、Conpot、Glastopf、Honeyd 等）的默认配置和行为特征
2. 每条规则设定 `confidence` 等级（HIGH/MEDIUM/LOW）
3. HIGH 置信度规则匹配特定标志（如 Banner 含 "cowrie" 关键字）
4. MEDIUM/LOW 置信度规则匹配常见版本特征或端口模式（如 Apache 老旧版本、非标准端口）
5. 资产级别取最高置信度检测结果

**教训**：蜜罐检测是一个持续完善的过程。37 条规则只是起点，实际部署后需要根据反馈不断调整规则库。建议后续添加规则管理 UI（启用/禁用/自定义）。

---

#### 难点 8：Hibernate 懒加载与 @Transactional 的边界

**问题**：`ReportService` 中 `hd.getRule()` 抛出 `LazyInitializationException`。原因是 `HoneypotDetection.rule` 关联为 `FetchType.LAZY`，而 `open-in-view=false` 导致 Hibernate Session 在 Repository 查询完成后即关闭。

**解决过程**：
1. 初始排查：curl 测试发现 taskId=1（有蜜罐数据）的 Excel 导出报 500，taskId=2（无蜜罐数据）正常
2. 定位到 `hd.getRule() != null ? hd.getRule().getRuleName() : ""` — Hibernate 代理对象非 null，但方法调用时 Session 已关闭
3. 添加 `@Transactional(readOnly = true)` 到 `generatePdfReport()` 和 `generateExcelReport()`

**教训**：`FetchType.LAZY` + `open-in-view=false` 组合下，访问懒加载关联必须在 `@Transactional` 作用域内。不能假设 getter 返回 null 意味着数据不存在 — Hibernate 代理对象永远非 null。

---

#### 难点 9：前后端鉴权边界

**问题**：扫描任务详情页导出按钮使用原生 `<a href>` 链接，浏览器导航时不携带 JWT `Authorization` header，导致 403 Forbidden。

**解决过程**：
1. 用户反馈 "Whitelabel Error Page" → 排查发现是 403 Forbidden
2. 对比 ReportCenterPage（使用 axios API 函数导出正常工作）与 ScanTaskDetailPage（使用 `<a>` 链接）
3. 将 `<a>` 改为 `<button>` + `onClick`，通过 axios 发出带 token 的请求

**教训**：所有需要鉴权的 API 调用必须通过 axios 拦截器（自动附加 token），不能使用原生 HTML 元素直接访问。前后端分离架构下，"能直接打开的链接"和"需要鉴权的下载"是两种不同的模式。

### 5.2 此前已总结的困难（继承）

1. **i18n 国际化持久战** — 200+ 硬编码中文替换，嵌套 JSX 表达式处理
2. **RSA 前后端协调** — JSEncrypt 与 Java Cipher 标准差异
3. **API 限流** — Bucket4j 过于复杂，自实现滑动窗口更轻量
4. **扫描进程管理** — ProcessRegistry + destroyForcibly()
5. **Flyway 与 Hibernate 共存** — 开发阶段禁用 Flyway，生产阶段切换

## 七、结语

ServerScout 项目从零开始，经历十一轮迭代，构建了一个功能完整的服务器资产攻击面可视化分析平台。本轮在已有完善功能基础上，新增了**蜜罐识别**这一差异化安全能力，解决了 PDF 中文乱码的顽疾，并完成了资产详情和爬虫面板的全面国际化。

**本轮核心贡献**：
1. **蜜罐识别系统**：37 条指纹规则覆盖 10+ 类别蜜罐框架，离线可检测，结果贯穿仪表盘→资产列表→资产详情→导出报告全链路
2. **PDF 中文渲染**：嵌入 Microsoft YaHei 字体，解决 iText 8 中文空白问题
3. **全面 i18n**：资产详情 + 爬虫面板新增 50+ 翻译键，总翻译键数达到 270+
4. **导出鉴权修复**：扫描任务详情页导出从 403 白屏到正常下载

技术选型上，采用 React 18 + TypeScript + Vite + Tailwind CSS + Spring Boot 3 + Java 17 + MySQL 全栈架构。集成了 Nmap/Nuclei 等工业级安全工具，通过策略模式实现扫描引擎的可扩展性，通过双层插件系统（L1 前端组件插件 + L2 后端扫描策略插件）实现功能灵活扩展。
