## 三、技术架构总览

```
展示层:  React 18 + TypeScript + Vite + Tailwind CSS
         ├── ECharts 5 (图表) + AntV G6 (拓扑V2)
         ├── React Query 5 (状态管理) + React Router 6 (路由)
         ├── react-i18next (国际化，270+ 翻译键)
         ├── JSEncrypt (RSA 前端加密)
         ├── Lucide React (图标)
         └── 内置用户手册页面 (ManualPage)

API 层:   Spring Boot 3.3.5 + Spring Security + JWT
         ├── 13 个 Controller / 26 个 Service
         ├── RSA 密钥对管理 / 请求限流拦截器
         ├── SSE 实时推送 / @Async 异步扫描
         └── 策略模式扫描引擎（Nmap + Nuclei + CustomCommand + ProcessRegistry）

数据层:  MySQL + Spring Data JPA + Flyway (就绪)
         ├── 17 个实体 / 17 个 Repository
         └── Hibernate ddl-auto: update (开发)

外部工具: Nmap (端口扫描) + Nuclei (漏洞检测) + Playwright (截图)
         + Shodan InternetDB + NVD CVE API + AlienVault OTX
         + Censys Search v2 + VirusTotal v3 + EPSS API + ip-api.com (IP归属地)
```

## 四、仍需优化的方向

### 4.1 功能层面

| 优先级 | 项目 | 描述 |
|:---:|------|------|
| 高 | 大规模扫描性能 | 当前单线程异步扫描，1000+ IP 耗时长，需线程池并发 |
| 高 | CVE 匹配精度 | 基于版本字符串简单包含，误报率较高，需语义化版本比较 + CPE 匹配 |
| 高 | 截图稳定性 | 依赖外部工具（Playwright/wkhtmltoimage/cutycapt），Windows 下不稳定 |
| 高 | 蜜罐检测完善 | 当前仅实现指纹规则离线检测，未集成 Nuclei 蜜罐模板在线检测；规则库可进一步扩充（如 T-Pot、Hfish 等现代蜜罐框架） |
| 中 | 蜜罐检测误报优化 | 部分规则（如 SSH 非标准端口、Redis 默认端口）置信度较低，需结合多维度特征交叉验证 |
| 中 | 资产去重逻辑 | 仅按 IP 去重，同 IP 不同域名可能被合并 |
| 中 | 外部情报缓存 | 内存缓存 10min TTL，重启丢失，建议引入 Redis |
| 中 | PDF 报告图表 | 当前 PDF 仅有固定表格布局，缺少 ECharts 图表嵌入（需服务端渲染方案） |
| 中 | 用户手册离线访问 | 当前手册页面加载截图依赖 public 目录静态资源，可打包为离线 PDF 内嵌 |
| 低 | 暗色模式细节 | 部分组件（G6 拓扑图、ECharts 图表、手册页面）暗色适配不完全 |
| 低 | 移动端体验 | 响应式已实现，但复杂页面（拓扑图、攻击面地图、手册侧边目录）移动端体验需优化 |

## 五、开发困难与教训总结

### 5.1 本轮新增困难

#### 难点 10：在 React 应用中内嵌完整用户手册

**问题**：用户手册内容量大（7章、16张截图、多个表格和代码块），如何在前端应用中优雅地呈现而不影响性能？

**解决过程**：
1. 首先编写了完整的 Markdown 手册文档（[ServerScout用户使用手册.md](docs/ServerScout用户使用手册.md)）
2. 考虑到 Markdown 需要额外依赖（react-markdown + rehype/remark 插件链），决定直接编写 React 组件
3. 复用 Tailwind CSS 的 typography 样式系统，自定义 `H2`/`H3`/`CodeBlock`/`Table`/`Img` 等渲染辅助函数
4. 左侧实现粘性目录导航（sticky TOC），点击触发 `scrollIntoView({ behavior: 'smooth' })`
5. 截图资源复制到 `frontend/public/docs/images/` 通过相对路径引用

**教训**：
- 内容型页面优先考虑 Markdown → 静态 HTML 的方案，React 组件适合交互型页面
- 大页面的目录导航应使用 `IntersectionObserver` 实现滚动自动高亮（当前仅实现了点击跳转）
- 如果手册频繁更新，应考虑 CMS 或文档生成工具（如 Docusaurus/VitePress）

---

#### 难点 11：前端静态资源路径管理

**问题**：手册页面中的截图需要在开发环境和生产构建中都能正确加载。Vite 的 `public/` 目录在开发时直接提供静态文件服务，但需要确保路径一致。

**解决过程**：
1. 截图放入 `frontend/public/docs/images/`
2. 组件中使用 `/docs/images/xxx.png` 绝对路径引用（Vite 开发和生产均从 public 根目录提供）
3. Markdown 文档中使用相对路径 `images/xxx.png`

**教训**：Vite 的 public 目录规则简单明了（直接复制到 dist 根目录），但团队新成员可能混淆 `src/assets/`（需 import）和 `public/`（直接 URL 访问）的区别。

---

### 5.2 此前已总结的困难（继承自 v12.0 及之前版本）

1. **iText 8 中文字体嵌入** — TTC 字体索引格式 `path,0`、`cPara()` 辅助方法统一管理
2. **蜜罐规则设计** — 低误报 vs 广覆盖平衡，37条规则覆盖10+类别
3. **Hibernate 懒加载与 @Transactional** — `open-in-view=false` 下访问懒加载关联必须在事务作用域内
4. **前后端鉴权边界** — `<a>` 链接不携带 JWT token，必须通过 axios 拦截器
5. **i18n 国际化持久战** — 270+ 翻译键，嵌套 JSX 表达式处理
6. **RSA 前后端协调** — JSEncrypt 与 Java Cipher 标准差异
7. **API 限流** — Bucket4j 过于复杂，自实现滑动窗口更轻量
8. **扫描进程管理** — ProcessRegistry + destroyForcibly()
9. **Flyway 与 Hibernate 共存** — 开发阶段禁用 Flyway，生产阶段切换

## 七、结语

ServerScout 项目从零开始，经历**十二轮迭代**，构建了一个功能完整的服务器资产攻击面可视化分析平台。

### 本轮核心贡献

1. **用户使用手册**：7章完整内容，面向不同角色的分众化文档，17张操作截图形成完整操作流
2. **内置帮助系统**：前端 ManualPage 组件集成，侧边栏一键访问，降低用户学习成本
3. **中英双语支持**：导航标签支持中英切换

### 项目亮点回顾

- **全链路蜜罐识别**：37条规则 → 仪表盘统计 → 资产角标 → 详情面板 → PDF/Excel报告
- **双模式扫描引擎**：策略模式 + L2插件系统，支持 Nmap/Nuclei/自定义命令
- **多维度可视化**：仪表盘(ECharts) + 拓扑图(G6) + 攻击面地图(D3.js)
- **完整的 DevOps 集成**：Webhook 通知 + 邮件报告 + 定时扫描
- **全面国际化**：270+ 翻译键覆盖全系统
- **安全加固**：RSA加密传输 + JWT认证 + 滑动窗口限流 + 前端错误监控

### 技术栈

> **前端**：React 18 + TypeScript + Vite + Tailwind CSS + Ant Design + ECharts + G6 + D3.js
> **后端**：Spring Boot 3 + Java 17 + MySQL + Spring Security + JWT + JPA
> **外部工具**：Nmap + Nuclei + Playwright + Censys + VirusTotal + Shodan + NVD
> **代码总量**：~39,800 行
> **开发周期**：2026-05-19 ~ 2026-05-24（6天12轮迭代）
