## 二、第九轮新增内容（本次迭代）

### 2.1 问题驱动的改进

本轮迭代围绕 6 个具体问题进行改进，覆盖了功能可见性、用户体验、配置灵活性等多个维度。

#### 问题 1：扫描策略插件 (L2) 和插件 (L1) 使用地点不明确

**现状**：L2 插件（自定义扫描策略）管理在设置页，但创建扫描任务时未体现；L1 前端插件虽标记了 PLUGIN 标签，但使用场景单一。

**改进**：
- **前端** `ScanTaskListPage.tsx`：扫描类型下拉框通过 `fetchScanTypes()` API 动态加载 L2 插件定义的扫描类型，标注「自定义策略」
- 已验证：`ssh-brute`（SSH弱口令检测）和 `custom` 等 L2 插件类型出现在下拉框中
- L1 插件 `PluginSlot` 已在仪表盘（`dashboard-widget`）和资产详情页（`asset-detail-top`）两个插槽渲染

#### 问题 2：IP 地址应显示真实归属地

**现状**：操作日志仅显示 IP 地址，无地理信息。

**改进**：
- **新增** `IpGeoService.java`：通过 ip-api.com 免费 API 查询 IP 归属地（国家/地区/城市/ISP），内网地址自动识别为「内网地址」
- **修改** `OperationLog.java`：新增 `geoLocation` 字段（VARCHAR 128）
- **修改** `OperationLogService.java`：异步解析 IP 归属地并存库
- **前端** `OperationLogViewer.tsx`：IP 列新增归属地显示（MapPin 图标 + 位置文字）
- 已验证：`127.0.0.1` 显示为「内网地址」

#### 问题 3：告警通知缺少邮件功能 + 外部情报 API 配置重复

**现状**：仅支持 Webhook（钉钉/飞书/企微），外部情报 API 密钥分散在告警通知配置中，有独立章节但不可直接编辑。

**改进**：
- **新增** `EmailNotificationService.java`：动态 SMTP 配置，HTML 格式邮件，异步发送
- **新增** `pom.xml` 依赖：`spring-boot-starter-mail`

- **修改** `SettingsPage.tsx`：
  - 外部情报 API 配置章节可直接编辑 Censys/VirusTotal 密钥（独立保存按钮）
  - 告警通知 Webhook 章节移除了重复的 API 密钥字段
  - 新增「邮件通知配置」章节：启用开关、收件邮箱、SMTP 服务器地址、端口、发件账号、授权码、SSL 开关

#### 问题 4：PDF 报告过于粗糙，信息映射不清晰

**现状**：PDF 仅有简单的表格，无数据说明和图例。

**改进** `ReportService.java`：

### 2.2 本轮补充修复（用户反馈）

在初步实现后，用户测试发现了三个额外问题：

#### 补充修复 1：SMTP 配置从网页输入

**问题**：邮件配置提示"通过 application.yml 配置"但实际应用中用户需要 Web UI 配置。

**改进** `EmailNotificationService.java`：
- 移除对 Spring 自动注入 `JavaMailSender` 的依赖
- 改为 `createMailSender()` 方法从数据库 SystemConfig 动态读取 SMTP 参数
- SMTP 服务器地址、端口、用户名、密码、SSL 开关 全部通过 Web UI 配置

#### 补充修复 2：ssh-brute 扫描类型无法正常工作

**问题**：`CustomCommandScanner` 仅返回漏洞结果，不返回资产。导致选择 `ssh-brute` 类型扫描后无资产入库。

**改进** `ScanExecutionService.java`：
- 当检测到策略为 `CustomCommandScanner` 且返回空资产列表时
- 自动先执行 Quick Nmap 扫描获取资产/端口
- 然后将 Nmap 资产 + 自定义插件漏洞合并为最终结果

#### 补充修复 3：邮箱通知开关状态保存问题

**问题**：邮件配置表单使用 `defaultChecked`/`defaultValue`（仅初始渲染生效），状态无法持久化。

**改进** `SettingsPage.tsx`：
- 全部改为 React `useState` 受控组件（`checked`/`value` + `onChange`）
- 使用 `useMutation` 管理保存操作
- 添加 loading 状态和错误处理

#### 补充修复 4：SMTP SSL/TLS 配置修复

**问题**：163.com 邮箱要求 SSL 与 STARTTLS 互斥，同开导致连接失败。
**报错**：`553 Mail from must equal authorized user`

**改进** `EmailNotificationService.java`：
- SSL 启用时自动禁用 STARTTLS，配置 SSLSocketFactory
- 添加 `helper.setFrom()` 确保发件地址与 SMTP 认证账号一致

**测试结果**：在浏览器中配置 163.com SMTP（端口 465，SSL 开启），创建扫描任务后成功发送邮件通知。

## 四、开发过程中的困难与教训

### 4.1 技术难点

#### 难点 1：CVE 匹配误报率高

**问题**：简单的版本字符串包含匹配导致大量误报。例如 `Apache 2.4.49` 匹配到 `CVE-xxx (Apache 2.x)` 但实际 CVE 受影响版本是 `2.2.x`。

**解决过程**：
1. 初版使用 `String.contains()` —— 误报率 > 50%
2. 引入版本范围解析（`<`, `>`, `>=`, `<=` 操作符）
3. 添加软件名称白名单（WEB_APP_KEYWORDS）和黑名单过滤（SKIP_CVE_AFFECTED）
4. 限制仅对 Web 服务端口进行 CVE 匹配

**教训**：漏洞关联是一个复杂的领域问题，简单的字符串匹配远不够。工业级方案应使用 CPE（Common Platform Enumeration）标准匹配 + NVD CPE Match Feed API。

#### 难点 2：截图功能稳定性

**问题**：Playwright Node.js 脚本调用不稳定，超时、进程残留、中文路径等问题频发。

**解决过程**：
1. 实现三级回退链：Playwright → wkhtmltoimage → cutycapt
2. 添加 35 秒超时和 `destroyForcibly()` 强制终止
3. 异步流式读取 stdout/stderr 避免管道阻塞
4. PNG 文件头校验（0x89PNG）确保输出有效

**教训**：调用外部进程时，超时控制、流式 I/O、错误处理缺一不可。中文路径问题在 Windows 开发中尤其需要注意。

#### 难点 3：自定义扫描策略与资产发现解耦

**问题**：`CustomCommandScanner` 只返回漏洞数据不返回资产，导致自定义策略扫描后无资产入库。

**解决过程**：设计两阶段扫描——先 Quick Nmap 发现资产，再自定义命令检测漏洞，合并两者结果。保证自定义插件既能看到漏洞也能看到资产。

**教训**：策略模式设计时需要考虑不同策略的能力边界（有些只做漏洞检测，有些只做资产发现），在编排层进行结果合并。

#### 难点 4：Windows 下 Java 版本管理

**问题**：系统默认 Java 8，项目需要 Java 17。Maven 使用 Java 8 编译导致 `ClassNotFoundException` 和 Spring Boot 启动失败。

**解决过程**：明确设置 `JAVA_HOME` 指向 JDK 17，使用 `mvn clean package -DskipTests` 打 JAR 包，再用 `java -jar` 直接启动绕过 `mvn spring-boot:run` 的类加载问题。

**教训**：Windows 多 JDK 环境下，不要依赖 `mvn spring-boot:run`（它在 fork 的 JVM 中运行，可能找不到 classpath），建议打 JAR 包后用绝对路径 Java 可执行文件启动。

#### 难点 5：SMTP 邮件发送调试

**问题**：邮件配置看似正确但始终发送失败，报错 `553 Mail from must equal authorized user`。

**解决过程**：
1. 发现 `From` 地址未设置，SMTP 服务器使用默认值导致与认证用户不一致
2. SSL/TLS 配置互斥问题：163.com 端口 465 需要纯 SSL，不能同时启用 STARTTLS
3. 添加 `mail.smtp.ssl.enable` 与 `mail.smtp.starttls.enable` 互斥逻辑

**教训**：邮件发送的坑主要在 SMTP 服务器的认证策略差异。QQ邮箱/163/Gmail 各有不同要求，需要提供灵活的配置项并在文档中说明。

### 4.2 工程实践教训

1. **前端状态管理**：`defaultChecked`/`defaultValue` 是非受控组件属性，仅在首次渲染生效。任何需要保存的表单都应使用 `useState` + `checked`/`value` 受控模式。

2. **API 设计**：将多个配置项（Webhook + API Key + Email SMTP）混在同一个 `updateSystemConfigs` 调用中会导致耦合。各自独立的保存按钮 + 独立的状态管理是更好的设计。

3. **测试驱动开发的价值**：本轮修改在浏览器中手动验证，重复多次"编码 → 启动服务 → 浏览器测试 → 修复"循环。如果有 Playwright E2E 测试，验证效率会大幅提升。

4. **数据库 Schema 变更**：添加 `enableCrawler`（ScanTask）和 `geoLocation`（OperationLog）列依赖 JPA `ddl-auto: update`，在开发阶段方便但生产环境应使用 Flyway/Liquibase 管理迁移。

## 六、结语

ServerScout 项目从零开始，经历九轮迭代，构建了一个功能完整的攻击面可视化分析平台。涵盖了从资产发现、端口扫描、Web 指纹识别、CVE 漏洞匹配、外部威胁情报查询、网络拓扑可视化、PDF/Excel 报告导出、邮件/Webhook 通知、用户权限管理、操作审计日志到截图爬虫的全链路功能。

技术选型上，采用 React + Spring Boot 全栈架构，集成了 Nmap/Nuclei 等工业级安全工具，通过策略模式实现扫描引擎的可扩展性，通过插件系统（L1 前端组件插件 + L2 后端扫描策略插件）实现功能扩展。

项目整体达到了毕业设计的预期目标：一个可运行、可演示、可扩展的网络安全工具平台。
