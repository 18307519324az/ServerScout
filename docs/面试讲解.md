# 面试讲解

## 一分钟项目介绍

ServerScout 是一个基于 Spring Boot 和 React 构建的服务器资产攻击面管理与风险分析平台，集成了 Nmap 和 Nuclei，用于完成资产发现、端口与服务识别、漏洞检测、攻击面可视化、报告导出和 AI 风险简报生成。这个项目主要体现我在后端服务设计、前后端协作、扫描工具集成和安全数据展示方面的工程实践能力。

## 三分钟项目介绍

项目的核心目标是把分散的安全扫描结果整合成统一平台。后端使用 Spring Boot，负责扫描任务创建、扫描编排、结果聚合、JWT 认证、统一异常处理和报告导出；前端使用 React + TypeScript，负责资产列表、漏洞列表、拓扑可视化、报告中心和 AI 风险简报等界面展示。  
在扫描能力上，我通过 Nmap 完成端口与服务探测，通过 Nuclei 完成漏洞模板检测，再把结果和 CVE、CVSS、EPSS 等信息结合，形成可查看、可导出、可解释的风险数据。  
项目还支持 PDF / Excel 报告导出，并通过 AI 风险简报模块把技术扫描结果转化成更容易阅读的风险摘要，这一部分更适合展示产品化思路和结果表达能力。

## 项目技术栈介绍

- 后端：Java 17、Spring Boot 3、Spring Security、JWT、JPA / Hibernate
- 前端：React 18、TypeScript、Vite、Tailwind CSS、Ant Design
- 数据库：MySQL、Redis
- 扫描能力：Nmap、Nuclei
- 可视化：ECharts、G6、D3.js
- 报告导出：PDF、Excel

## 核心业务流程

1. 用户创建扫描任务并输入目标范围。
2. 后端校验任务参数并进入扫描执行流程。
3. 后端调用 Nmap 获取端口与服务信息。
4. 后端调用 Nuclei 获取漏洞检测结果。
5. 后端聚合资产、端口、漏洞、证书、指纹与情报数据。
6. 前端展示资产、漏洞、拓扑和报告视图。
7. 用户导出报告或生成 AI 风险简报。

## 后端难点

- 扫描任务的执行编排与状态维护
- Nmap / Nuclei 外部工具调用与结果解析
- 资产、漏洞、端口、证书、指纹等数据之间的关联建模
- 统一异常处理与安全响应设计
- 报表导出中的中文字体与多维数据组织

## 前端难点

- 多页面状态管理与接口交互组织
- 资产、漏洞、拓扑等多类型数据的展示一致性
- 图表与图结构可视化的交互实现
- 鉴权、错误处理和手册页面等通用体验设计

## 扫描工具集成难点

- 不同工具输出格式不同，需要统一解析
- 需要控制超时、并发和错误处理
- 工具安装路径在不同系统中的适配方式不同
- 结果不能只停留在原始输出，必须映射到可查询的数据结构中

## 数据库设计思路

- 使用 `Asset` 作为资产主实体
- 使用 `Port` 表示端口与服务信息
- 使用 `AssetVulnerability` 将资产与漏洞关联
- 使用 `ScanTask` 管理任务生命周期
- 使用 `ScanAssetMapping` 支持任务与资产关系跟踪

## 权限认证设计

- 基于 Spring Security 和 JWT
- 登录后发放 Token
- 通过角色区分管理员与普通用户能力
- 对关键接口进行访问保护

## 项目亮点

- 兼顾扫描、分析、展示、导出和说明能力
- 安全工具集成不是简单调用，而是做了结构化结果沉淀
- 同时覆盖后端设计、前端展示和文档工程化
- 适合作为毕业设计或全栈安全项目案例

## 项目不足

- 工具链依赖本地环境安装，跨平台一致性仍需提升
- 更复杂的任务调度与插件体系仍可继续完善
- 部分模块的自动化测试覆盖还可以继续补强

## 后续优化方向

- 增加更多扫描模板和自动化联动
- 优化任务调度、权限模型和多租户隔离
- 扩展情报接入和可视化维度
- 增强 AI 风险简报的上下文理解与模板输出

## 常见面试问答

### 为什么要集成 Nmap 和 Nuclei？

Nmap 适合做端口和服务探测，Nuclei 适合做漏洞模板检测，两者结合后可以覆盖“发现资产”和“识别风险”两个关键阶段。

### 为什么要做统一异常处理？

统一异常处理可以保证前后端交互结构一致，也能避免把底层异常直接暴露给前端，提升系统可维护性和安全性。

### AI 风险简报的价值是什么？

它的价值不在替代扫描，而在于把技术扫描结果整理成更易读的风险摘要，降低结果解释成本。
