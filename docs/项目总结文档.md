## 1. 项目概述

**ServerScout** — 服务器资产攻击面可视化分析平台，面向中小型企业安全运维人员，提供从资产发现、指纹识别到漏洞关联的全流程攻击面管理能力。

### 1.1 核心价值

- **轻量易部署**：基于 Spring Boot 3 + React 18，开箱即用
- **可视化优先**：仪表盘、拓扑图、攻击面地图、雷达图等直观展示
- **外部情报集成**：Shodan InternetDB + NVD CVE + AlienVault OTX + EPSS
- **自动化扫描**：Nmap + Nuclei + 子域名枚举 + SSL 证书采集 + CVE 匹配

### 1.2 参考项目

| 项目 | 借鉴内容 |
|------|---------|
| **reNgine** (7K+ stars) | 暗色模式 UI、自动化侦察流水线、定时扫描 |
| **IVRE** (4K+ stars) | Nmap XML 解析、被动监控概念 |
| **ProjectDiscovery / Nuclei** (22K+ stars) | 模板化漏洞扫描、CVE 匹配引擎 |
| **Sn1per** (8K+ stars) | 多模式扫描预设 (Quick/Stealth/Web/Full) |
| **Keygraph / Shannon** (40K+ stars) | Pipeline 可视化、攻击面地图、技术栈雷达 |

## 4. 新增功能详解（本轮迭代）

### 4.1 暗色模式（Dark Mode）

**借鉴来源**：reNgine 的暗色主题 UI

**实现方式**：
- Tailwind `darkMode: 'class'` 策略
- CSS 自定义属性定义 light/dark 色彩体系
- `useDarkMode` Hook 管理主题状态（persist 到 localStorage，跟随系统偏好）
- Layout 侧边栏月亮/太阳切换按钮

**影响范围**：Layout、Dashboard、ExternalIntel 等所有主要页面

### 4.2 风险评分系统

**评分公式**：`风险分 = 严重×10 + 高危×5 + 中危×2 + 低危×1`

**风险等级**：
- 严重（≥100）→ 红色
- 高（≥50）→ 橙色
- 中（≥20）→ 黄色
- 低（<20）→ 绿色

### 4.3 扫描模板预设（Sn1per 风格）

| 预设 | 扫描类型 | 端口范围 | 指纹识别 | 漏洞扫描 |
|------|---------|---------|:---:|:---:|
| **Quick** | quick | 1-1000 | ✅ | - |
| **Stealth** | quick | 1-1000 | ✅ | - |
| **Web** | quick | 80,443,8080,8443,3000,5000,7000 | ✅ | ✅ |
| **Full** | full | 1-65535 | ✅ | ✅ |

### 4.4 Pipeline 可视化（Shannon 风格）

5 阶段流水线在扫描运行时实时展示：

```
Recon → Fingerprint → Vuln Scan → CVE Match → Report
(端口扫描)  (Web指纹)   (漏洞扫描)   (CVE匹配)   (报告)
```

每个阶段根据实际进度自动切换状态：等待 → 进行中（旋转动画）→ 已完成（绿色勾）

### 4.5 攻击面地图

**借鉴来源**：Shannon 的 Reconnaissance 层级图谱

**层级结构**：
```
Attack Surface
  └─ 192.168.1.0/24
       ├─ 192.168.1.1 (web-server)
       │    ├─ :80 http
       │    │    ├─ nginx 1.18
       │    │    └─ CVE-2021-23017 [HIGH]

       │         └─ CVE-2022-1292 [CRITICAL]
       └─ 192.168.1.2 (db-server)
```

使用 ECharts Tree 图实现，支持展开/折叠，节点按类型着色。

## 6. 遇到的困难与教训

### 6.1 困难 1：后端未启动导致前端"假死"

**现象**：仪表盘永远显示"加载中..."，没有错误提示。

**原因**：React Query 在 API 返回 500 时 `isLoading` 变为 `false`，但 `data` 为 `undefined`，原代码 `isLoading || !stats` 导致始终显示 loading 状态，用户无法感知错误。

**解决**：添加 `isError` 状态检测，区分「加载中」和「加载失败」，失败时展示错误信息 + 重试按钮。

**教训**：前端状态处理必须覆盖 **loading / success / error** 三个状态。不能只判断"不是成功就是加载中"。

### 6.2 困难 2：Java 版本不匹配

**现象**：Spring Boot 3.3.5 JAR 启动时报 `UnsupportedClassVersionError`。

**原因**：系统默认 Java 为 1.8（class version 52），但 Spring Boot 3 需要 Java 17（class version 61）。

**解决**：使用 JDK 17 的完整路径启动：`"C:/Program Files/Java/jdk-17/bin/java" -jar target/server-scout-1.0.0.jar`

**教训**：
- 项目应在 README 中明确声明 Java 版本要求
- 考虑使用 Maven Wrapper + `JAVA_HOME` 环境变量来统一 Java 版本
- CI/CD 中应锁定 JDK 版本

### 6.3 困难 3：TypeScript 类型安全与 any 的博弈

**现象**：ECharts 配置对象中动态索引 `severityColor[s.name]` 触发 TS7053 错误。

**原因**：`s.name` 类型为 `any`，无法用于索引 `Record<string, string>`。

**解决**：用 `as Record<string, string>` 类型断言。

**教训**：外部库（ECharts）的类型定义不够精确时，需要在类型安全和开发效率之间权衡。建议对频繁使用的图表配置抽取为类型安全的工厂函数。

### 6.4 困难 4：前后端路径不一致

**现象**：curl 测试时用 `/api/v1/auth/login` 返回 403，以为是 CSRF 配置问题。

**原因**：AuthController 映射在 `/api/auth`，前端 axios baseURL 是 `/api`，实际路径是 `/api/auth/login`，而非 `/api/v1/auth/login`。

**教训**：API 路径规范要统一，建议所有业务 API 统一在 `/api/v1/` 下，或者 auth 也统一前缀。

### 6.5 教训总结

1. **先启动后端再调前端**：前端调试的第一步是确认后端是否可达
2. **前后端 API 路径需统一规范**：本次 auth 路径不统一导致排查耗时
3. **错误状态必须有用户可见的反馈**：不能让 UI 进入"永远加载"的死循环
4. **构建前检查 Java 版本**：`mvn package` 成功不代表 `java -jar` 能运行
5. **借鉴开源项目要先理解其核心思路**：Shannon 的 Pipeline、reNgine 的 Dark Mode、Sn1per 的 Scan Presets 都是优秀的 UX 模式
