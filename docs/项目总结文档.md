## 1. 项目概述

**ServerScout** — 服务器资产攻击面可视化分析平台，面向中小型企业安全运维人员，提供从资产发现、指纹识别到漏洞关联的全流程攻击面管理能力。

### 1.1 核心价值

- **轻量易部署**：基于 Spring Boot 3 + React 18，开箱即用
- **可视化优先**：仪表盘、拓扑图、攻击面地图、技术栈雷达图等直观展示
- **外部情报集成**：Shodan InternetDB + NVD CVE + AlienVault OTX + EPSS
- **自动化扫描**：Nmap + Nuclei + 子域名枚举 + SSL 证书采集 + CVE 匹配

## 4. 本轮迭代新增（2026-05-22）

### 4.1 日志噪音过滤

**问题**：外部扫描器（bot/scanner）频繁探测 `index.php`、`wp-admin` 等不存在路径，`NoResourceFoundException` 被通用异常处理器以 ERROR 级别记录完整堆栈，日志被大量无效信息淹没。

**修复**：在 `GlobalExceptionHandler` 中新增 `NoResourceFoundException` 专用处理器，以 DEBUG 级别记录，返回简洁 404 响应。

**影响文件**：[GlobalExceptionHandler.java](serverscout/backend/src/main/java/com/serverscout/util/GlobalExceptionHandler.java)

### 4.2 同 IP 多主机名管理

**问题**：同一 IP（如 127.0.0.1、192.168.x.x）在不同环境下可能有不同主机名，重新扫描时会**覆盖**旧主机名，无法追溯历史命名。

**修复**：
- `Asset` 实体新增 `hostname_aliases` JSON 列，存储所有出现过的主机名
- `ScanExecutionService.saveScanResults()` 改为累积模式（`appendHostname()`），不覆盖
- `AssetResponse` DTO 新增 `hostnameAliases` 字段
- `AttackSurfaceService` 构建节点时展示别名

**影响文件**：

### 4.4 修复建议格式化 + 内容细化

**问题**：
1. 修复建议被拼为单行字符串，页面显示混乱难读
2. 大量 CVE 的修复建议过于空泛（"升级到修复版本"）

**修复**：

**前端格式化**：新增 `formatSteps()` 函数，按 `1)` `2)` `3)` 模式拆分建议文本，渲染为 `<ol>` 有序列表，每步独立一行。

**后端内容细化**：将约 40 个 CVE 的修复建议从单行升级为多步骤操作指南，包含：
- 具体的检查命令（`httpd -v`、`find / -name`）
- 精确的版本号（`2.4.51+`、`5.7.2+`）
- 日志检查路径（`/var/log/audit`、`access.log`）
- 临时缓解措施（WAF 规则、禁用模块）

**示例对比**：
- 之前：`"升级到修复版本"`
- 之后：
  1. 限制 TMUI 访问来源 IP
  2. 升级到 15.1.0.4+/14.1.2.6+/13.1.3.4+
  3. 检查 /var/log/audit 日志确认无异常访问

**CveDataInitializer 升级**：从"仅插入新记录"改为"插入 + 更新已有记录"，确保历史数据也被增强。

**影响文件**：
- [CveDataInitializer.java](serverscout/backend/src/main/java/com/serverscout/config/CveDataInitializer.java) — 76 条种子数据中的修复建议全部细化

## 6. 遇到的困难与教训

### 6.1 困难 1：Hibernate LazyInitializationException

**现象**：攻击面地图 API 返回 500，日志显示 `could not initialize proxy [CveDatabase#1] - no Session`。

**原因**：`spring.jpa.open-in-view: false` 关闭了 Hibernate 的 OSIV，Session 在 Service 层返回后即关闭。`AttackSurfaceService` 没有 `@Transactional`，导致在 Controller 层（视图渲染时）访问懒加载的 `cveDatabase` 代理失败。

**解决**：
1. 给 `buildAttackSurfaceMap()` 添加 `@Transactional(readOnly = true)`
2. 在 Repository 中使用 `JOIN FETCH` 预加载关联数据

**教训**：每当设置 `open-in-view: false`（推荐做法）时，必须确保所有读操作在 `@Transactional` 范围内完成懒加载。

---

### 6.2 困难 2：JSX 属性中的 Unicode 转义

**现象**：VulnerabilityDetailPage.tsx 编译失败，Babel 报 `Expecting Unicode escape sequence \uXXXX`。

**原因**：`placeholder` 属性中包含了用户示例字符串 `{\"key\":\"value\"}`，Babel JSX 解析器将 `\"` 中的 `\u` 序列误识别为 Unicode 转义。

**解决**：将 placeholder 改为 JavaScript 表达式语法（`placeholder={"..."}`），避免 JSX 属性字符串中的转义问题。

**教训**：在 JSX 属性值中包含特殊字符（`\`、`"`、换行）时，优先使用 `{}` 表达式语法，避免 JSX 字符串解析歧义。

---

### 6.3 困难 3：CVE 修复建议更新不生效

**现象**：后端日志显示 `CVE seed: 76 total, 0 newly added`，前端看到的修复建议仍是旧版本。

**原因**：`CveDataInitializer` 仅在数据库中不存在该 CVE 时才插入（`findByCveId().isEmpty()`），已存在的记录不会被更新。

**解决**：修改初始化逻辑，对已存在的 CVE 也比较修复建议长度，若新的更长则更新。本迭代从"48 updated"可确认生效。

**教训**：种子数据初始化器应考虑升级场景，不能假设数据库为空。应支持"插入新 + 更新旧"模式。

---

### 6.4 困难 4：前端编译错误阻塞 E2E 测试

**现象**：Vite 编译错误遮罩覆盖整个页面，Playwright 无法进行任何交互。

**原因**：前端的 babel 编译错误在开发模式以全屏 overlay 展示，阻止所有点击事件。

**解决**：先修复编译错误再继续测试。

**教训**：在开发流程中应先确认编译无误（`npm run build` 或观察终端 Vite 输出），再进行浏览器测试。

---

### 6.5 困难 5：Java 运行时版本不匹配

**现象**：`mvn spring-boot:run` 报 `ClassNotFoundException`，但 `mvn compile` 成功。

**原因**：系统默认 Java 8 与 Spring Boot 3.3.5 不兼容。

**解决**：使用 `"C:/Program Files/Java/jdk-17/bin/java" -jar` 显式指定 JDK 17 运行。

**教训**：
- 应在项目根目录配置 `.java-version` 或 `JAVA_HOME` 明确的脚本
- `mvn compile` 成功不代表能运行，运行时也需要相同 Java 版本

---

### 6.6 教训总结

1. **OSIV 关闭后所有读操作必须加 @Transactional**：否则懒加载必炸
2. **JOIN FETCH 优于依赖懒加载**：N+1 查询 + LazyInit 异常双杀
3. **种子数据要支持增量更新**：不能假设每次都是空库启动
4. **JSX 属性有特殊字符用 `{}` 表达式**：比字符串更安全
5. **先修编译错误再测浏览器**：Vite overlay 会阻断所有 E2E 操作
6. **锁定运行时 Java 版本**：编译通过不代表能运行
7. **前端状态要同时处理 loading/error/empty 三种状态**：不能只判断"不是成功就是加载中"
8. **UI 修复建议必须分行显示**：单行长文本在卡片中不可读
