## 五、开发困难与教训总结

### 5.1 核心技术难点

#### 难点 1：i18n 国际化是一场持久战

**问题**：全项目 200+ 处硬编码中文，翻译工作量大且容易遗漏。JSX 表达式中的中文（如 `{var ? '已启用' : '已禁用'}`）不能简单地用 `replace_all` 批量替换，会形成嵌套 `t()` 调用。

**解决过程**：
1. 先用 `replace_all` 批量替换明显的中文字符串
2. 手动修复嵌套 `t()` — 如 `{var ? "{t('settings.enabled')}" : "{t('settings.disabled')}"}` → `{var ? t('settings.enabled') : t('settings.disabled')}`
3. 多轮浏览器测试发现遗漏
4. 子组件（ExternalIntelPage 内的 IpResultCard、CveSearchResults 等）需各自添加 `useTranslation()` hook
5. 参数化文本如 `共 {count} 条` 最初使用 i18n 插值语法失败，改用 `.replace()` 方案

**教训**：
- i18n 应在一开始就建立规范，所有文本通过 `t()` 调用，禁止硬编码
- 不要假设 `replace_all` 能安全处理所有场景，JSX 表达式内需要特别注意
- 子组件需要各自引入 `useTranslation()`，不能依赖父组件传递 `t` 函数

---

#### 难点 2：RSA 加密的前后端协调

**问题**：JSEncrypt 与 Java `Cipher` 的标准差异。后端用 `Cipher.getInstance("RSA")` 加密/解密，前端用 JSEncrypt。

**解决过程**：
1. 后端生成 2048 位密钥对，Base64 编码公钥通过 API 下发
2. JSEncrypt 接收 PKCS#8 格式公钥
3. `encryptor.encrypt()` 返回 `string | false`，需做类型守卫处理

**教训**：RSA 加密在 Web 场景下是一个标准方案，但 TypeScript 类型安全要求对 `false` 返回值做显式处理。

---

#### 难点 3：API 限流的粒度选择

**问题**：Bucket4j 引入后与 Spring Boot 3 有兼容问题，且增加了不必要的复杂度。

**解决过程**：放弃 Bucket4j，改用自定义 `ConcurrentHashMap` 滑动窗口实现，不到 80 行代码解决问题。

**教训**：简单的限流需求不需要引入外部库。`ConcurrentHashMap` + 时间窗口 = 够用且可控。

---

#### 难点 4：扫描进程管理

**问题**：取消扫描任务后，操作系统级的 nmap/nuclei 子进程仍在运行。

**解决过程**：
1. 创建 `ProcessRegistry` 使用 `ConcurrentHashMap<Long, List<Process>>`
2. 每个扫描器（NmapScannerImpl、NucleiScannerImpl、CustomCommandScanner）在 `pb.start()` 后注册 Process
3. 取消时调用 `process.destroyForcibly()`，并在 finally 块中 `cleanup()`

**教训**：Java 的 `Process.destroy()` 是优雅终止（SIGTERM），`destroyForcibly()` 是强制终止。对于 nmap/nuclei 这类外部工具，`destroyForcibly()` 更可靠。

---

#### 难点 5：Flyway 与 Hibernate ddl-auto 的共存

**问题**：Flyway 的 V2 迁移脚本使用了 `ADD COLUMN IF NOT EXISTS`（MySQL 不支持），导致迁移失败。

**解决过程**：
1. 初期尝试修复 SQL 语法
2. 发现开发阶段 ddl-auto: update 已自动创建了列
3. 最终方案：开发阶段 `flyway.enabled: false` + `ddl-auto: update`，生产部署时切换

**教训**：Flyway 的引入应考虑现有数据库状态。已有数据的项目应先导出基线脚本（V1），再增量迁移。

---

### 5.2 工程实践教训

1. **JSX 中的 `replace_all` 需谨慎**：中文在 JSX 属性值、表达式、模板字符串中各有不同的替换策略，没有一刀切的方案。

2. **翻译键的命名规范**：使用 `模块.功能` 二级命名（如 `settings.pluginName`），避免与 login/common 等全局键冲突。

3. **参数化文本的处理**：i18next 的 `{{variable}}` 插值与 JSX 花括号语法冲突，`.replace()` 虽然"脏"但可靠。

4. **子组件翻译独立性**：每个使用翻译的组件都应有自己的 `useTranslation()` hook，不要通过 props 传递 `t` 函数。

5. **浏览器测试不可替代**：编译通过不代表 i18n 正确。多次在浏览器切换语言后发现遗漏，说明自动化测试或至少 Playwright 快照测试是必要的。

6. **依赖选择应遵循"够用就好"**：Bucket4j 被自定义限流器替代，Flyway 在开发阶段禁用——新依赖引入前应评估实际需求。

## 七、结语

ServerScout 项目从零开始，经历十轮迭代，构建了一个功能完整的服务器资产攻击面可视化分析平台。项目涵盖了资产发现、端口扫描、Web 指纹识别、CVE 漏洞匹配、外部威胁情报查询、网络拓扑可视化、PDF/Excel 报告导出、邮件/Webhook 通知、用户权限管理、操作审计日志、截图爬虫、国际化双语支持的全链路功能。

技术选型上，采用 React 18 + TypeScript + Vite + Tailwind CSS + Spring Boot 3 + Java 17 + MySQL 全栈架构。集成了 Nmap/Nuclei 等工业级安全工具，通过策略模式实现扫描引擎的可扩展性，通过双层插件系统（L1 前端组件插件 + L2 后端扫描策略插件）实现功能灵活扩展。
